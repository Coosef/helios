# Helios — Sprint 1 Implementation Plan

- **Status:** Active (implementation in progress)
- **Date:** 2026-06-08
- **Owner:** Engineering, under Chief Architect
- **Authority chain:** CLAUDE.md → PRD → ADR-001..004 → 02-TECHNICAL-DESIGN (incl. §0 binding decisions) → 04-ACCEPTANCE-CRITERIA → 03-TASK-BREAKDOWN.

This plan governs the implementation of **Sprint 1 (Agent Foundation) only**. No future-sprint functionality (backup, restore, compression, encryption engines, SaaS UI/DB) is implemented; only the Sprint-1 reservations called out in the ADRs are produced.

## Guardrails (non-negotiable, from CLAUDE.md)

- Work **sprint by sprint**; never implement future-sprint features.
- **Security > speed**; **restore correctness > backup speed.**
- **No hardcoded secrets**, **no encryption/signing private keys in source**; only the **public** update-signing test key is committed.
- **No placeholder security**: the updater's signature/hash/anti-rollback verification is **real and enforcing** (ADR-002); "placeholder" = stubbed orchestration only.
- English identifiers and comments; structured logging; Clean Architecture + SOLID + DI.
- **Tests for every new unit**; **complete files only**, never partial snippets.

## Per-task workflow

For every task the same five steps are performed and recorded:

1. **Explain** what is being implemented and which design section / acceptance criteria / risks it satisfies.
2. **Generate complete files** (production code).
3. **Generate tests** (unit and/or integration as the task warrants).
4. **Update documentation** (this plan's status tracker, and any affected docs).
5. **Mark task status** in the tracker below.

## T14 Enrollment Recovery Forward Items

The S1-T14 enrollment recovery review (2026-06-09) confirmed the use-case is correct for its scope and **does not block merge**; the following recovery/robustness items were deferred. They are owned by later tasks / Sprint 2, not by the T14 code.

### FI-1 — Sprint 2 / Medium — Atomic enrollment persistence

- `state.Store` must support a **transactional/batch write** so the enrollment credential is persisted **atomically** (Technical Design §0.2: "Persist credential atomically, go ENROLLED"). Today the values are written per-key, so a crash mid-sequence can leave a partial state.
- The **T19 enrolled-state predicate** must require **all** of:
  - `device_id` present,
  - `agent_certificate_pem` present,
  - `agent_session_token` present.
- `device_id` alone must **never** mean ENROLLED (it is written first; a partial write would otherwise read as enrolled and then 401 on first heartbeat).

### FI-2 — Sprint 2 / Medium — Crash-safe enrollment recovery

- **Persist and reuse the `Idempotency-Key`** across a restart for the **same logical enrollment attempt** (currently a new key is generated per attempt and not persisted).
- The **server must replay the original `201` response** on a matching `Idempotency-Key` (so a crash between server success and local persist is self-healing rather than stranding the device).
- The **re-enrollment token flow must reuse `device_guid` and the license seat** (ADR-003 decision 6; §0.2/597) — the recovery path when the single-use token is already consumed (replay → `409`, fatal by design, §0.2/493).

### FI-3 — T18 / T29 / Low — Enrollment token delivery cleanup

- **Reconcile** the shipped `BEYZ_ENROLLMENT_TOKEN` env approach with the designed **one-shot `enroll.token` file** (ACL-locked; Technical Design §0.2/897/1251).
- Implement **delete-on-success** of the one-shot file.
- **Zero the in-memory token** after success where possible.
- **Never persist** the enrollment token (the agent already satisfies this; the gap is the source-artifact lifecycle in the installer/service layers).
