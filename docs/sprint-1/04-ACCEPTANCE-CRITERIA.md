# Beyz Backup — Sprint 1 Acceptance Criteria

> Measurable, objectively-verifiable exit conditions. **Must-have** = Sprint-1 exit blocker. **Stretch** = desirable, not blocking. Each has an explicit verification method.

**42 criteria** — 38 must-have, 4 stretch.

## Must-have (Sprint-1 exit blockers)

| ID | Category | Criterion | Verification |
|----|----------|-----------|--------------|
| AC-01 | Service | beyz-backup-agent.exe installs, registers, and starts as a Windows Service named BeyzBackupAgent with start type auto and account LocalSystem. | Run installer (or `beyz-backup-agent.exe install`), then `sc qc BeyzBackupAgent` shows START_TYPE=AUTO_START and SERVICE_START_NAME=LocalSystem; `sc query BeyzBackupAgent` reports STATE=RUNNING. |
| AC-02 | Service | The service supports the full SCM lifecycle: stop, start, restart, and uninstall, transitioning cleanly with no orphaned process. | Pester/`sc.exe` sequence: `sc stop` -> STOPPED; `sc start` -> RUNNING; restart -> RUNNING; uninstall -> `sc query` returns 1060 (service does not exist). After each step `tasklist` confirms exactly one or zero agent processes. |
| AC-03 | Service | The agent survives a reboot and an unexpected process kill: it auto-starts after reboot and is restarted by configured Windows Service recovery actions after a crash. | After reboot, `sc query BeyzBackupAgent` is RUNNING. `sc qc BeyzBackupAgent`/`sc failure` shows restart-on-failure configured (e.g. 5s/30s). Kill the PID with `taskkill /F`; within the reset window `sc query` returns to RUNNING and state files remain intact. |
| AC-04 | Service | The agent runs in console mode for development and as a service in production, selecting mode via flag, with graceful shutdown and panic recovery (no crash-loop on a recovered panic). | `beyz-backup-agent.exe run --console` runs in foreground and exits 0 on Ctrl+C; unit test injects a panic in a worker and asserts recover() logs the panic and the run loop continues; service stop completes within the SCM timeout (graceful shutdown logged). |
| AC-05 | Config | The agent loads config.yaml from C:\ProgramData\BeyzBackup\config.yaml, validates it against the committed JSON Schema (config.schema.json), and on missing or schema-invalid config exits non-zero with an actionable error naming the offending key. | Unit/integration test: valid sample loads and exits 0; delete the file -> non-zero exit + error mentioning the path; corrupt a required field (e.g. `server.url=http://...`) -> non-zero exit + error naming `server.url`. No panic/stack trace as the user-facing error. |
| AC-06 | Config | Configuration precedence is deterministic: CLI flag overrides environment variable (prefix BEYZ_) overrides config.yaml overrides compiled default. | Table-driven unit test sets the same key at multiple layers (default, file, `BEYZ_LOGGING_LEVEL`, `--log-level`) and asserts the effective value follows flag > env > file > default for at least 3 keys. |
| AC-07 | Config | The config parser carries a config_version integer, ignores unknown/reserved fields without error (forward-compatibility), and the reserved future keys (crypto.wrapped_device_key, key_wrap_version, key_id, storage, backup) parse to null/empty without breaking load. | Unit test feeds config.sample.yaml plus an extra unknown key and the reserved blocks; load succeeds, reserved fields are null/zero-value, and an unrecognized future key produces no error. A config_version newer than supported is rejected with a clear message. |
| AC-08 | Config | Secret material is structurally absent from config.yaml: no token, private key, license, or storage credential field exists in the schema or sample, and the parsed config struct exposes no secret values. | Static check: `config.schema.json` and `config.sample.yaml` contain no token/key/secret/password fields (grep). Unit test marshals the loaded config to JSON and asserts no secret-named field is present. |
| AC-09 | Logging | All operational log output is single-line JSON (zerolog) written to C:\ProgramData\BeyzBackup\logs\agent.log, and every record carries ts (RFC3339 UTC), level, component, event, and agent/device identifier. | Run the agent; assert every line in agent.log parses as standalone JSON and contains the required fields. Automated test pipes log output through a JSON line-parser and fails on any non-JSON or missing-field line. |
| AC-10 | Logging | Security/audit events are emitted to a separate stream (security.log) using the frozen schema {schema_version, seq, prev_hash, hash, ts_local, event_type, tenant_id, device_id, actor, outcome}, with per-record BLAKE3 hash chaining, for enrollment and update events. | Trigger enroll and update events; assert security.log records match the frozen schema and that each record's hash = BLAKE3(prev_hash || canonical(event)). Unit test mutates one record and asserts chain verification fails. |
| AC-11 | Logging | Log rotation is enforced (lumberjack) by size/age/backup count, and security events are always written regardless of operational log level (lowering level cannot silence the audit trail). | Unit/integration test writes past max_size_mb and asserts a rotated backup file is created and old backups are pruned to max_backups. Set logging.level=error and assert an info-severity enrollment security event still appears in security.log. |
| AC-12 | Security | No secret value (enrollment token, private key, license blob, bearer/Authorization header, storage credential, proxy password) ever appears in any log stream or installer log, enforced by a redaction wrapper. | Run full enroll+heartbeat+install flow; grep agent.log, security.log, and the installer /LOG file for the known test token, key bytes, and bearer value -> zero matches. Unit test asserts the Secret wrapper's String()/MarshalJSON() returns ***REDACTED***. |
| AC-13 | Enrollment | On first run the agent generates and persists a random device GUID (UUIDv4) and a local keypair, builds a CSR, and the GUID survives an app reinstall that preserves ProgramData. | Unit test: GUID generated once, persisted to state, and identical across simulated restarts; regenerated only when the state file is absent. Integration: uninstall with state-keep then reinstall -> same device GUID. |
| AC-14 | Enrollment | The agent enrolls by calling POST /v1/enroll with the token in the request body (never URL/query), receives and atomically persists device_id, signed agent certificate, agent_session_token, and tenant_id to the protected state store. | Integration test against Prism mock seeded from openapi.yaml: agent enrolls, then state store contains agent.crt, device_id, and tenant_id; assert the HTTP request placed the token in the body, not the URL. State write uses write-temp->fsync->rename (verified by injecting a crash before rename and confirming no partial file). |
| AC-15 | Enrollment | Re-running enrollment with an already-consumed token fails: the mock returns 409, the agent surfaces a clear non-retried error, logs a security event, and does NOT overwrite or corrupt existing valid enrolled state. | Integration test enrolls successfully, then re-enrolls with the same token; assert 409 handled, `enrollment.token_rejected`/failure security event emitted, no blind retry loop, and existing agent.crt/device_id unchanged. |
| AC-16 | Enrollment | Device identity is the server-issued opaque device_id plus cert SPKI thumbprint; hardware signals (MachineGuid, disk serial) are reported advisory-only and are not used as the primary key or license key. | Code/test review: enrollment request includes hardware signals only in an advisory fingerprint block; identity resolution and license-seat binding key off device_id/cert, not hardware hash. Unit test asserts identity remains stable when simulated hardware signals change. |
| AC-17 | Enrollment | Secret material (agent private key, license blob) is stored encrypted at rest via Windows DPAPI machine scope inside the ACL-locked state store; unwrap of a tampered blob fails closed. | Unit test round-trips wrap->unwrap of key bytes via DPAPI (CRYPTPROTECT_LOCAL_MACHINE) and asserts a flipped-byte blob fails to unwrap and returns an error rather than garbage. Inspect on-disk key file is not plaintext PEM. |
| AC-18 | Heartbeat | After enrollment the agent sends POST /v1/agents/{id}/heartbeat with X-Agent-Version and X-Protocol-Version headers, carrying the minimal presence payload (version, status enum, health block) and not the heavier poll data. | Integration test against mock asserts heartbeat requests arrive carrying both version headers and a body limited to version/status/health fields; assert payload does not include task data. |
| AC-19 | Heartbeat | Heartbeat cadence is server-controlled: the agent applies cadence.next_heartbeat_seconds from the response with mandatory +/- jitter (default 20%), using the config value only as a pre-first-response fallback floor. | Mock returns next_heartbeat_seconds=N; integration test measures inter-heartbeat intervals and asserts they fall within N +/- jitter%, and differ from the config fallback. Unit test verifies jitter is applied and never zero. |
| AC-20 | Heartbeat | On heartbeat errors the agent applies exponential backoff with full jitter capped at backoff_max_seconds, continues local operation while degraded (no hard crash), and handles 401 (->re-enroll/degraded) and 426 (->update-required, stop calling) distinctly. | Integration test: mock returns 5xx -> assert increasing backoff intervals capped at the ceiling and agent stays alive; mock returns 426 -> agent logs update_required and stops normal calls; mock returns 401 -> agent transitions to degraded/re-enroll path. Unit test asserts backoff cap and full-jitter distribution. |
| AC-21 | TaskPolling | The agent polls GET /v1/agents/{id}/tasks on the server-supplied jittered cadence, parses the versioned task envelope, logs 'no tasks' for the empty placeholder, and applies next_poll_seconds from the response. | Integration test against mock returning tasks:[] and next_poll_seconds=N: assert a 'no tasks' log line, that polling honors N +/- jitter, and that the request carries version headers. |
| AC-22 | TaskPolling | The task-poll and heartbeat clients tolerate unknown/reserved fields (work_available, hold_seconds, rollout fields, schema_version) without erroring, preserving forward compatibility. | Unit/contract test feeds responses containing extra/reserved fields and future schema_version values; assert parsing succeeds, work_available is honored when true (immediate poll), and unknown fields are ignored. |
| AC-23 | Updater | beyz-backup-updater.exe verifies a real Ed25519 signature over the canonical (RFC 8785 JCS) update manifest against a public-key SET embedded at build time, selecting the key by manifest key_id; verification is enforcing code, never a return-true stub. | Unit test with fixtures: valid signature -> accept; flipped manifest byte -> reject; unknown key_id -> reject; revoked key_id (in manifest revocation list) -> reject. Source review confirms no `return true` shortcut and the embedded key is a public key only. |
| AC-24 | Updater | The updater reads the expected binary hash only from the verified manifest and verifies SHA256 (and BLAKE3) of the exact downloaded bytes before any swap; a hash mismatch aborts the update before replacing the binary. | Integration test: serve a binary whose bytes do not match the manifest hash -> assert update aborts at the hash step, no swap occurs, `update.hash_mismatch` logged, and staged file discarded. Source review confirms the hash never comes from a header/filename. |
| AC-25 | Updater | The updater enforces anti-rollback: it refuses any update whose signed manifest target_version is <= the persisted current_version (and below min_supported_version), unless a separately-signed emergency-downgrade flag is present. | Unit/integration test: manifest with target_version <= current -> rejected with `update.downgrade_blocked`; manifest below floor -> rejected. Version compared from signed manifest, not filename/header (verified by test serving a higher filename but lower manifest version). |
| AC-26 | Updater | The updater performs the staged atomic replace: stop BeyzBackupAgent service, snapshot current binary+config to backup, MoveFileEx(.new -> live, REPLACE_EXISTING) atomic rename, restart service, against a fixture manifest. | Integration test drives the FSM against a fixture (valid-signed) manifest and asserts the sequence stop->backup->atomic-rename->start occurred (state transitions logged/persisted) and that an interrupted rename leaves either old or new binary, never a truncated one. |
| AC-27 | Updater | The post-update health gate triggers rollback: if the new agent fails to start or does not self-report 'update OK' via heartbeat within the bounded timeout (90s), the updater integrity-checks the backup, restores the prior binary+config pair, restarts, and emits update.rolled_back. | Integration test serves a validly-signed bad binary that exits without heartbeating; assert health gate times out at 90s, backup integrity is checked (BLAKE3 == recorded) before restore, prior binary/config restored, service back to RUNNING, and `update.rolled_back` security event emitted. |
| AC-28 | Updater | The updater FSM state is persisted to state (write-temp-rename) before each side-effecting step so a crash/power loss mid-update is recoverable: on restart it resumes or rolls back from the persisted state. | Integration test injects an updater process kill in a post-SWAPPED state, restarts the updater, and asserts it reads persisted fsm_state and either completes or rolls back to a consistent (binary,config) pair without bricking. |
| AC-29 | Updater | The updater rejects a badly-signed or spoofed manifest before any download trust is granted: no binary is staged or swapped when signature verification fails. | Integration test serves a manifest with an invalid/forged signature; assert the updater never downloads/stages/swaps, logs `update.signature_invalid`, and the live binary is unchanged. |
| AC-30 | Installer | The Inno Setup installer lays down both binaries to C:\Program Files\BeyzBackup, creates the ProgramData folder tree (config.yaml, state\, logs\, update\), and registers + starts the SINGLE BeyzBackupAgent service post-install — the updater binary is installed but NOT registered as a second service (§0.6). | Run `setup.exe /VERYSILENT`; assert both .exe files exist under Program Files, the ProgramData subfolders exist, `sc query BeyzBackupAgent` is RUNNING, `sc query BeyzBackupUpdater` returns 1060 (no second service registered), and the installer exit code is 0. |
| AC-31 | Installer | The installer sets restrictive ACLs at folder-create time: inheritance broken and Users removed on state\ (SYSTEM + Administrators only); logs\ is Users-read; config.yaml is Users-read, Admin/SYSTEM write. | After silent install, automated `icacls C:\ProgramData\BeyzBackup\state` shows only SYSTEM and Administrators (no Users/Authenticated Users) and `/inheritance:r` applied; `icacls` on logs\ and config.yaml match the documented matrix. |
| AC-32 | Installer | The installer accepts the enrollment token both via an interactive masked wizard page and via the /TOKEN= (silent) parameter, hands it to first-run enrollment via a transient ACL-locked one-shot file that is deleted on consume, and never writes the token to config.yaml. | Silent: `setup.exe /VERYSILENT /TOKEN=<test>` -> agent enrolls against mock and the one-shot token file is gone afterward; config.yaml contains no token. | AC-33 | Installer | The enrollment token value never appears in the installer log when using /LOG. | Run `setup.exe /VERYSILENT /TOKEN=<test> /LOG=install.log`; grep install.log for the token value -> zero matches. |
| AC-34 | Security | The agent control-channel HTTP client enforces TLS 1.2 minimum (1.3 preferred), validates the server certificate, and pins the SaaS SPKI: a connection to a server presenting a non-pinned certificate is refused. | Integration test points the agent at an httptest server with a different SPKI than the configured pin -> connection refused/error; assert system-CA-only trust is rejected for the control channel and `spki_pin.mismatch` logged. | AC-35 | Security | No secrets or private keys are committed to the repository or CI: only the update test PUBLIC key is embedded, and a secret-scanning gate (gitleaks/trufflehog) is a required CI check. | CI runs gitleaks and fails the build on any committed secret; repo grep confirms build/keys contains only a public PEM (no private key); the update private key lives in the CI secret manager, not the repo. |
| AC-36 | Testing | CI runs go vet, golangci-lint (incl. gosec/staticcheck), unit tests, and contract tests against the Prism mock generated from the committed openapi.yaml, all green, with the agent client generated from the spec (drift fails the build). | GitHub Actions run shows `go test ./...` green, lint passes, contract tests pass against Prism, and a regenerate-from-OpenAPI step detects no drift. ## Stretch (non-blocking)

| ID | Category | Criterion | Verification |
|----|----------|-----------|--------------|
| AC-39 | Build | The build cross-compiles the agent and updater for windows/amd64 plus linux/amd64 and linux/arm64 using a pinned Go toolchain, and a lint-clean systemd unit file is produced for the Linux prep (not required to run in CI). | Taskfile `cross` target produces all four binaries; `go version` matches the pinned toolchain in go.mod; the systemd unit passes `systemd-analyze verify` (or equivalent lint) in CI. |
| AC-40 | Installer | Both binaries and the installer .exe are Authenticode-signed, separate from the Ed25519 update trust root. | `signtool verify /pa beyz-backup-agent.exe`, the updater, and the setup .exe all report a valid Authenticode signature; documentation confirms update decisions rely on the Ed25519 manifest signature, not Authenticode. |
| AC-41 | Installer | Uninstall stops the BeyzBackupAgent service and removes any updater scheduled task, makes a best-effort de-enroll call to release the license seat, and crypto-shreds the state store by default (with /KEEPSTATE to preserve identity for device replacement). | Run uninstall: assert the BeyzBackupAgent service is stopped and removed (and confirm no second service ever existed), a `POST /deenroll` attempt observed at the mock, and the state\ secrets purged (no key/cert/license remain). With /KEEPSTATE, assert state is preserved and reused on reinstall. |
| AC-42 | Updater | (§0.6 scope-guard) The updater ships as an on-demand binary only: it is NOT registered as a persistent Windows service, and Sprint 1 contains NO updater-self-update / two-stage-bootstrap code and NO persistent-watchdog loop (both deferred to Sprint 8). | After install, `sc query BeyzBackupUpdater` returns 1060 (does not exist); invoking the updater runs the FSM and exits (no resident process remains between runs); source review confirms there is no updater-self-update bootstrap and no always-on watchdog loop in the Sprint-1 code. |

## Definition of Done (rollup)

Sprint 1 is **Done** when: every must-have criterion above passes; `go test ./...` and the required CI gates (`gosec`, `gitleaks`, `govulncheck`, manifest-signature negative tests) are green on the Windows + Linux build; the agent installs/enrolls/heartbeats/polls against the Prism mock; the updater verifies+swaps+rolls-back a fixture update and rejects a tampered/badly-signed one; the installer silently installs with `/TOKEN=`, sets locked ACLs, and starts the service; and the OpenAPI contract + the design records (DR-01..06) are committed.
