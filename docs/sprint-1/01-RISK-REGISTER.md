# Beyz Backup — Sprint 1 Risk Register

> Pre-implementation architecture risk review. Author: Lead Software Architect. Date: 2026-06-08.
> Severity = blast radius if unaddressed. Likelihood = chance it bites given the current docs. **S1** = must be *designed-for or decided during Sprint 1* (the agent foundation), even when the risky feature ships in a later sprint — because Sprint 1 freezes the agent↔server contract, on-disk format, identity scheme, and update trust root that auto-update makes impossible to flag-day-migrate later.

## Executive risk summary

- **85 risks** across 9 dimensions + 6 added by the completeness review (17 Critical, 41 High).
- **75 are Sprint-1-relevant** — they shape primitives Sprint 1 bakes in.
- **The dominant theme:** nearly every Critical risk is a *decide-now / impossible-later* foundational decision (encryption-key custody & recoverability, update-signing trust root, manifest format, tenant/identity binding, protocol versioning). The product can be built sprint-by-sprint, but these contracts **cannot** be retrofitted across a deployed, auto-updating fleet.
- **The single biggest structural risk** is sequencing (ARCH-1): Sprint 1 builds the agent against a SaaS API that does not exist until Sprint 2. Mitigation is a committed, versioned OpenAPI contract as the source of truth — but see REV-6 (a self-authored mock proves self-consistency, not Sprint-2 parity).

## 1. Architectural Risks

*Beyz Backup's documentation describes an ambitious multi-tenant, zero-knowledge, hybrid backup SaaS, but the architecture is defined at a dangerously high level of abstraction relative to the irreversibility of Sprint 1 decisions. The single most serious problem is sequencing: Sprint 1 builds the agent (enrollment, heartbeat, task-polling, registration) against a SaaS API that does not exist until Sprint 2, meaning the agent's entire wire protocol, auth model, and state machine will be invented blind and then frozen by the installer rollout. The 5-component model (SaaS / Agent / Update / Storage / Restore) is materially incomplete: it has no home for the message queue (Redis), the manifest/metadata database, key management, the licensing/billing engine, the update-signing infrastructure, or the certificate authority — all of which are cross-cutting and several of which (key custody, signing-key custody, fingerprint scheme, audit log, protocol versioning) are expensive-to-retrofit foundational decisions that must be made now even though the features ship later. Clean Architecture/SOLID is asserted but the zero-knowledge encryption boundary, multi-tenant isolation boundary, and agent/server contract are not specified, so "API-first" is currently a claim without an artifact. The risks below are concrete and tied to specific document gaps; the majority are Sprint-1-relevant because they shape the agent's on-disk and on-wire contracts that the installer will distribute to customer machines.*

### ARCH-4 — Update-signing key custody and trust root undefined while updater verifies signatures in Sprint 1
`Critical severity · Medium likelihood · **S1**`

- **Risk:** PRD req 12 and CLAUDE.md ('Do not remove update signature validation') require the Sprint 1 updater skeleton to verify an update signature; SECURITY.md says all updates must be signed and the agent must verify Signature + SHA256 + Version Manifest. But nothing defines where the signing private key lives, how the public key/trust root is embedded in the agent, the signature algorithm, or the key-rotation story. ROADMAP defers 'Signed Updates' to Sprint 8, creating a gap where the verifier exists but the trust root is unspecified.
- **Impact:** The update channel is the highest-value attack surface in any agent product — compromise of the signing key or trust root is full remote code execution as a Windows Service (SYSTEM) on every customer endpoint, i.e. a supply-chain catastrophe and a ransomware vector against a backup product. If the public-key pinning and algorithm are not baked into the Sprint 1 agent binary correctly, retrofitting the trust root means re-distributing every installer.
- **Recommendation:** Decide now: signature algorithm (e.g. Ed25519 or RSA-PSS over a canonical manifest), pin the public key (or a small set, for rotation) compiled into the agent, and document private-key custody (offline/HSM, not in CI, not in the repo). Implement the Sprint 1 updater to verify against a real embedded test public key — never a no-op stub that 'returns true' — so the verification path is exercised from day one. Specify key rotation (overlapping keys) before the first ship.

### ARCH-5 — Zero-knowledge encryption boundary asserted but key custody architecture absent — risks a non-zero-knowledge design
`Critical severity · High likelihood · **S1**`

- **Risk:** SECURITY.md mandates client-side encryption where 'Server must never access plaintext files' (zero-knowledge), and ROADMAP puts Key Management in Sprint 4. But there is no architecture for how the data-encryption key is derived, stored, escrowed, or recovered, nor how it relates to the agent certificate issued at enrollment. The tension is fundamental: true zero-knowledge means the provider cannot recover a customer's data if they lose their key, yet SMB/hotel/MSP customers will expect support to help them restore.
- **Impact:** If key management is designed in Sprint 4 without a Sprint-1 decision, the team will likely take the easy path of server-side key storage or server-mediated key derivation, silently breaking the zero-knowledge promise the product is marketed on (a compliance/legal exposure for a security product). Conversely a strict design with no recovery path produces permanent data loss and support disasters. Either way, the enrollment flow built in Sprint 1 (which issues the agent identity) must already anticipate where keys come from.
- **Recommendation:** Write a one-page key-management decision record NOW, even though code lands in Sprint 4: define the key hierarchy (per-tenant/per-device data key, how derived, e.g. from a customer passphrase/recovery key vs. provider-escrowed), the explicit zero-knowledge vs. recoverability trade-off, and an optional recovery-key escrow model the customer opts into. Ensure the Sprint 1 enrollment payload reserves fields for key/recovery material so the flow need not change later.

### ARCH-1 — Agent built in Sprint 1 against a SaaS API that does not exist until Sprint 2
`High severity · High likelihood · **S1**`

- **Risk:** ROADMAP.md sequences Sprint 1 (Agent Foundation: enrollment, heartbeat, task polling, SaaS registration placeholder) before Sprint 2 (SaaS Core: tenant/user/RBAC/agent dashboard). PRD requirements 7-10 require the agent to enroll, register, heartbeat, and poll tasks — all of which are HTTP calls to a server contract that has no owner yet. The 'placeholder' framing hides that the agent's request/response schemas, auth headers, error semantics, and retry behavior are being invented inside the agent with no server counterpart to validate against.
- **Impact:** The agent's wire protocol gets frozen by the Inno Setup installer rollout to customer machines. When Sprint 2 builds the real FastAPI server, mismatches in field names, auth scheme, status codes, or pagination force either a breaking agent change (re-distributing installers to every device in the field) or permanent server-side compatibility shims for a contract that was never reviewed. Multi-location/MSP customers with many agents make field re-rollout especially costly.
- **Recommendation:** Before writing agent HTTP code in Sprint 1, author a versioned OpenAPI 3 spec for the four endpoints the agent calls (POST /v1/enroll, POST /v1/agents/{id}/register, POST /v1/agents/{id}/heartbeat, GET /v1/agents/{id}/tasks). Generate the agent client from that spec and stub a contract-test server (e.g. Prism/Schemathesis) so the placeholder validates against the same artifact Sprint 2 will implement. Commit the spec to the repo as the source of truth, not the Go code.

### ARCH-2 — No explicit agent/server protocol version negotiation; auto-update guarantees a fleet of mixed versions
`High severity · High likelihood · **S1**`

- **Risk:** CLAUDE.md and PRD mandate agent auto-update, and ROADMAP defers signed updates to Sprint 8 — meaning agents will self-update across many server releases. Yet neither ARCHITECTURE.md nor SECURITY.md mentions an API version, capability negotiation, or a min-supported-agent policy. Endpoints in the docs are unversioned. With auto-update, the field will permanently contain agents of N different versions hitting one server.
- **Impact:** Without a negotiated version, a server change silently breaks older agents (failed heartbeats look like offline devices; failed task polls silently skip backups — a data-protection product failing silently is the worst outcome). Conversely the server can never safely evolve its schema. Retrofitting versioning after agents are deployed requires a flag day.
- **Recommendation:** Decide the versioning scheme NOW: put /v1/ in every path, send X-Agent-Version and X-Protocol-Version headers on every request, and have the server return 426/explicit upgrade-required when an agent is below the minimum supported version. Define the compatibility window policy (e.g. server supports agents up to 3 minor versions back) in SECURITY.md/ARCHITECTURE.md before the first installer ships.

### ARCH-3 — 5-component model omits queue, metadata DB, key management, billing, signing infra, and CA
`High severity · High likelihood · **S1**`

- **Risk:** ARCHITECTURE.md lists exactly five components (SaaS, Agent, Update Service, Storage, Restore Engine) but the stack in CLAUDE.md includes Redis (queue) and PostgreSQL, and the product requires: a manifest/restore-point metadata DB (referenced in data flow but unowned), client-side key management (Sprint 4, but custody model undecided), device + storage-quota licensing/billing, an update-signing service with private-key custody (Sprint 8), and a certificate authority issuing agent certs after enrollment (SECURITY.md 'Agent Certificate, generated after enrollment'). None of these appear as components or are assigned a home.
- **Impact:** Cross-cutting concerns with no architectural owner become accidental couplings: e.g. the manifest DB ends up half in the SaaS service and half in the Restore Engine; key management and the CA get bolted onto whatever service touches them first. Billing/licensing enforcement scattered across endpoints is hard to make consistent and auditable. The diagram understates true system complexity, leading to underestimated effort and missing boundaries.
- **Recommendation:** Expand ARCHITECTURE.md to name the missing components explicitly and assign ownership: (1) Task/Job Queue (Redis), (2) Metadata/Manifest Store (Postgres) with a documented schema owner, (3) Key Management boundary, (4) Licensing & Billing service, (5) Update Signing service (offline/HSM key custody), (6) Enrollment CA. Even if most are stubs until later sprints, the boundaries and data ownership must be drawn before the agent's contracts solidify in Sprint 1.

### ARCH-6 — Multi-tenant isolation boundary undefined; agent identity scheme set in Sprint 1 must encode tenancy
`High severity · Medium likelihood · **S1**`

- **Risk:** CLAUDE.md lists 'Multi-tenant first' as the top architecture principle and the product is a multi-tenant SaaS for MSPs (who manage many downstream tenants). Yet no document defines the tenancy model: shared-schema-with-tenant-id vs. schema/DB-per-tenant, how the agent's enrollment token and resulting certificate bind a device to a tenant, or how task-polling/heartbeat requests are scoped so one tenant's agent can never read another's tasks. The agent fingerprint and enrollment built in Sprint 1 are the root of this binding.
- **Impact:** Cross-tenant data leakage is an existential risk for an MSP-targeted backup product. If the Sprint 1 enrollment/fingerprint scheme doesn't carry an immutable tenant binding into the agent certificate, retrofitting tenant isolation later means re-enrolling the entire fleet. MSP nesting (one MSP, many client tenants) is unaddressed and is hard to add to a flat tenant model later.
- **Recommendation:** Decide the tenancy model and MSP hierarchy before enrollment is coded. Make the enrollment token tenant-scoped, embed an immutable tenant_id (and, for MSPs, a parent org id) into the agent certificate issued at enrollment, and require every agent request to be authorized against that bound tenant server-side. Document the isolation guarantee (row-level tenant_id filter vs. per-tenant schema) in ARCHITECTURE.md.

### ARCH-7 — Agent fingerprint and enrollment-token single-use semantics underspecified — re-enrollment and hardware-change breakage
`Medium severity · High likelihood · **S1**`

- **Risk:** SECURITY.md states the enrollment token is 'single-use only' and an agent certificate is generated after enrollment; PRD lists 'Agent fingerprint' as a security feature. But the fingerprint's inputs (MAC? machine GUID? disk serial? TPM?) and stability are undefined, and single-use tokens have no documented recovery path for legitimate re-enrollment (OS reinstall, disk swap, VM clone, hardware refresh). Device-based licensing (PRD) depends directly on a stable, non-spoofable fingerprint.
- **Impact:** If the fingerprint is unstable (e.g. MAC-based), routine IT events change it and break the device's identity, license seat, and backup continuity — a constant support burden for hotels/SMBs. If it's spoofable, device-count licensing is trivially evaded (revenue leakage). Single-use tokens with no re-enrollment flow mean every re-image requires manual admin intervention. These are baked into the Sprint 1 enrollment code.
- **Recommendation:** Define the fingerprint composition now: prefer a persisted, agent-generated random device GUID stored under C:\ProgramData\BeyzBackup (surviving app reinstall) combined with stable hardware signals, and document VM-clone handling. Specify token lifecycle explicitly (single-use + expiry + server-side revocation) and design a re-enrollment/device-replacement flow that reuses the license seat. Capture this in SECURITY.md before coding enrollment.

### ARCH-8 — Agent statefulness, config/secret layout, and crash-safe state are undefined for a Windows Service writing to ProgramData
`Medium severity · Medium likelihood · **S1**`

- **Risk:** Sprint 1 puts config.yaml and logs under C:\ProgramData\BeyzBackup, but the docs never address where mutable agent state lives: the issued agent certificate/private key, the device GUID, last-known task cursor, and update rollback state. CLAUDE.md forbids hardcoding secrets and mandates env vars for sensitive config, yet a Windows Service has no interactive env and the agent certificate's private key must persist on disk. There is also no story for config/secret separation or file ACLs on ProgramData (which is world-readable by default).
- **Impact:** If the agent's private key or enrollment material lands in a world-readable ProgramData path with default ACLs, any local user can exfiltrate the device identity (impersonation, tenant access). Mixing immutable config (config.yaml) with mutable runtime state in one file causes update/rollback races. The CLAUDE.md 'env vars for secrets' rule is unworkable for a service and will be silently violated, undermining the stated security posture.
- **Recommendation:** Define the on-disk layout now: immutable config.yaml (operator-editable) vs. a separate machine-protected state store for the cert/private key and device GUID, secured via Windows DPACI/DPAPI or a restricted ACL (SYSTEM + Administrators only), not env vars. Document this in SECURITY.md and reconcile the CLAUDE.md env-var rule to mean 'no secrets in source/config' rather than 'all secrets in env'. Make state writes atomic (write-temp-rename) for crash safety.

### ARCH-9 — Audit log / tamper-detection design deferred to Sprint 8 but must be schema-stable from Sprint 1
`Medium severity · Medium likelihood · **S1**`

- **Risk:** PRD lists 'Audit log' and 'Tamper detection' as security features and SECURITY.md requires all security events (enrollment, update, restore, failed integrity, auth failure) to be logged — yet ROADMAP defers Audit Logging to Sprint 8. Sprint 1 already produces the earliest auditable events (enrollment, update attempts) via generic structured logging with no defined audit schema, no event taxonomy, and no tamper-evidence (append-only/hash-chain) design.
- **Impact:** Audit logs are only trustworthy for compliance/incident response if they were captured consistently and tamper-evidently from the start. If Sprint 1 emits enrollment/update events as ad-hoc log lines, the Sprint 8 audit trail will have a blind spot for the most security-sensitive early lifecycle events, and retrofitting a tamper-evident schema means losing historical integrity. For a security product sold to MSPs, a credible audit trail is a sales/compliance requirement.
- **Recommendation:** Define a minimal structured audit-event schema now (event_type, tenant_id, device_id, timestamp, actor, outcome) and a shipping/retention target, and have Sprint 1 emit enrollment and update events through that schema even if the central audit store arrives in Sprint 8. Decide the tamper-evidence approach (server-side append-only with hash chaining) at design time so the event shape doesn't change later.

### ARCH-10 — 'API-first' principle has no artifact; Restore Engine and Storage Layer boundaries are undefined despite restore-correctness being a top priority
`Low severity · Medium likelihood`

- **Risk:** CLAUDE.md asserts 'API-first' and ranks 'Restore correctness first' above backup speed, but there is no API contract artifact (no OpenAPI, no schema) and the Restore Engine (Sprint 6) and Storage Layer are described only as bullet lists of responsibilities/targets with no interface defined. The Storage Layer abstracts five very different backends (SMB/SFTP/S3/MinIO/Beyz Cloud) with no common interface specified, and the Restore Engine's relationship to the manifest DB and the encryption/key boundary is unstated.
- **Impact:** Without a defined storage interface, each backend integration (Sprint 7) risks leaking backend-specific assumptions (e.g. S3 multipart semantics vs. SMB file locking) into the backup/restore engines, breaking the clean-architecture goal. 'API-first' without a committed spec tends to become 'API-after', where the API mirrors whatever the implementation happened to do. Restore correctness can't be designed-for if the restore path's contract with storage and keys is undefined.
- **Recommendation:** Commit to a single source-of-truth API spec repo artifact now (OpenAPI) and define a storage-backend Go interface (Put/Get/List/Resume/Delete chunk + capability flags) before Sprint 7, plus a restore contract that explicitly names its dependencies on the manifest store and key-management boundary. This makes the Clean Architecture/SOLID claim verifiable rather than aspirational.

## 2. Security Risks

*The Beyz Backup documents stake the entire product on "client-side encryption / zero-knowledge" and "device + storage-quota licensing," yet the foundational security decisions that make or break those promises are either unspecified or deferred past the point where they can be cheaply retrofitted. The single most dangerous gap is key management: ROADMAP defers "Key Management" to Sprint 4, but the agent built in Sprint 1 already defines the config layout, enrollment exchange, and identity model that any zero-knowledge key scheme must live inside — decide it now or rebuild the agent later. SECURITY.md is one page of aspirations ("Single-use only," "All updates must be signed") with no mechanism: no key derivation, no token format/TTL/transport, no signing-key custody, no audit-log integrity model, no threat model. The agent runs as SYSTEM/root with an auto-updater that replaces its own binary — a textbook supply-chain target — yet signing-key custody, update-channel pinning, and config/secret-on-disk protection are undefined. mTLS is deferred to Sprint 8, leaving Sprint 1 on plain "TLS + bearer token"; that is acceptable only if TLS is pinned and the token model is hardened now. Below are 9 concrete risks; the foundational ones (key-management origin, signing-key custody, enrollment token format, agent identity/fingerprint, config-secret-at-rest, audit-log integrity, protocol versioning) are flagged Sprint 1-relevant because they are architectural and expensive to change after the agent and installer ship to customer machines.*

### SEC-1 — Zero-knowledge key origin/derivation/recovery is undefined and deferred to Sprint 4, but the agent foundation locks in the identity and config model it must use
`Critical severity · High likelihood · **S1**`

- **Risk:** PRD and CLAUDE.md promise 'Secure client-side encryption' / zero-knowledge (SECURITY.md: 'Server must never access plaintext files'), and ROADMAP pushes 'Key Management' to Sprint 4 under Compression & Encryption. Nowhere is it defined where the per-tenant/per-device data encryption key comes from: is it derived from a customer passphrase (Argon2/scrypt), generated on-device and wrapped, or — fatally — issued by the SaaS server? If the server ever sees or escrows the key, the zero-knowledge claim is false and a SaaS breach exposes all customer backups. If the key is purely on-device with no recovery path, a single disk failure or reinstall makes every backup permanently unrecoverable — catastrophic for a backup product whose entire value is restore. CLAUDE.md itself ranks 'Restore reliability higher than backup speed,' yet no key-recovery design exists. The Sprint 1 agent already defines config.yaml layout, the enrollment handshake, and the device identity that any key scheme must bind to.
- **Impact:** Either a server-side breach decrypts every tenant's backups (zero-knowledge promise broken, contractual/GDPR catastrophe), or customers permanently lose all backups on key loss with no recovery — both existential for a backup vendor. Retrofitting a key model after agents are deployed means re-enrolling and re-keying every device in the field.
- **Recommendation:** Before writing Sprint 1 config/enrollment code, decide and document the key hierarchy: (1) generate a per-device data key on-device; wrap it with a tenant Key-Encryption-Key derived from a customer-held recovery passphrase via Argon2id; (2) store only the wrapped key, never plaintext, and never transmit the KEK or unwrapped DEK to the server (zero-knowledge); (3) define an explicit, tested recovery flow — printed/escrowed recovery code with optional opt-in M-of-N or customer-managed escrow — and surface in config.yaml placeholder fields (e.g. wrapped_device_key, key_wrap_version) so the on-disk format needs no breaking change in Sprint 4. Write this into SECURITY.md now as the authoritative key-management spec.

### SEC-3 — Update signing-key custody is completely unspecified — the highest-value key in the system protects a SYSTEM-level auto-updater
`Critical severity · Medium likelihood · **S1**`

- **Risk:** SECURITY.md ('All updates must be signed') and CLAUDE.md ('Do not remove update signature validation') mandate signed updates, and the Sprint 1 updater skeleton already implements 'signature verification' (PRD req 12). But nothing states where the signing private key lives, who can use it, or how it is protected. The agent runs as a Windows Service (SYSTEM) and the updater stops the service, replaces the binary, and restarts it. Whoever controls the signing key controls code execution as SYSTEM on every customer endpoint. If that key sits on a CI server, a developer laptop, or a cloud secret store without hardware backing, one compromise = mass remote code execution across all tenants (a SolarWinds/3CX-class supply-chain event). Even though signed updates are 'hardened' in Sprint 8, the updater that consumes the signature and the public key it trusts are pinned into the binary in Sprint 1.
- **Impact:** Compromise of the signing key yields persistent SYSTEM-level RCE on every deployed endpoint across every tenant — the worst possible outcome for a security product and unrecoverable for the company's reputation.
- **Recommendation:** Decide custody before shipping the updater that trusts a public key: keep the update-signing private key in an HSM or offline air-gapped signing ceremony (or at minimum a cloud HSM/KMS with hardware backing, strict IAM, and audit), never on CI runners or laptops. Embed the corresponding public key (or a pinned key-set with a documented rotation/rollover plan) in the agent at build time in Sprint 1, and design the manifest to carry a key-id so keys can rotate without re-issuing every agent. Require signing to be a multi-person, logged action.

### SEC-2 — Enrollment token has no specified format, TTL, transport, or replay/brute-force protection beyond the word 'single-use'
`High severity · High likelihood · **S1**`

- **Risk:** SECURITY.md says only 'Enrollment Token — Single-use only.' PRD requirement 18 lets the installer accept the token via 'enrollment token input or command-line parameter.' There is no defined entropy/length, no expiry/TTL, no binding to a tenant or device, no rate limiting on the registration endpoint, and no statement of how single-use is enforced (server-side atomic consumption vs. client honesty). A command-line token on Windows is exposed to any local process via the process table and lands in shell history, Inno Setup logs, and unattended-install scripts. Short or guessable tokens invite brute force against the registration placeholder. 'Single-use' enforced only client-side is no enforcement at all.
- **Impact:** A leaked or brute-forced enrollment token lets an attacker enroll a rogue agent into a victim tenant — receiving backup policies, storage targets, and (depending on SEC-1) key material, enabling data theft or fleet poisoning. Tokens in CLI args/logs are a realistic leakage vector on shared/MSP-managed machines.
- **Recommendation:** Specify now: token = >=128 bits CSPRNG, single-use enforced by atomic server-side consumption (DB row state, not client trust), short TTL (e.g. 15-60 min), bound to a tenant_id and optionally a pre-registered device fingerprint. Prefer stdin/secure-file/installer dialog over a plaintext CLI arg; if CLI is supported, zero it from memory and exclude it from installer logs. Rate-limit and lock the registration endpoint, and log every enrollment attempt (success/fail) per SECURITY.md. Define all of this in Sprint 1 since the agent registration call and installer parameter are being built now.

### SEC-4 — Plain TLS + bearer token for Sprint 1 with no certificate pinning leaves enrollment and heartbeat open to MITM until mTLS lands in Sprint 8
`High severity · Medium likelihood · **S1**`

- **Risk:** mTLS is deferred to Sprint 8 (ROADMAP) and SECURITY.md lists it only under 'Future Security Features,' so Sprint 1 enrollment/registration, heartbeat, and task-polling rely on plain TLS plus a token/agent certificate. Nothing specifies TLS certificate or public-key pinning. SMBs, hotels, and MSP environments frequently run TLS-intercepting middleboxes, captive portals, and corporate proxies that terminate TLS. Without pinning, a middlebox or attacker with a trusted-CA cert can intercept the enrollment exchange (capturing the enrollment token and any returned credentials/agent cert) and the heartbeat/task channel (injecting forged tasks once a command channel exists).
- **Impact:** MITM during enrollment leaks the token and the issued agent credential, letting an attacker impersonate the device or hijack its task channel; in later sprints this becomes a path to injecting malicious restore/backup commands to a SYSTEM-level agent. The targeted environments (hotels, multi-location SMBs via MSPs) are exactly where TLS interception is common.
- **Recommendation:** For Sprint 1, plain TLS+token is acceptable ONLY if you pin the SaaS server's certificate or public key (SPKI pin) in the agent, reject system-CA-only trust for the control channel, and require TLS 1.2+/1.3. Treat the agent certificate issued at enrollment as a client credential and lay the groundwork to promote it to true mTLS in Sprint 8 without changing the enrollment format. Document the pin-rotation procedure now.

### SEC-5 — Agent certificate / fingerprint scheme is named but not designed — issuance, private-key storage, and what the fingerprint binds to are undefined
`High severity · Medium likelihood · **S1**`

- **Risk:** SECURITY.md: 'Agent Certificate — Generated after enrollment.' PRD lists 'Agent fingerprint' as a security feature. But it is unspecified whether the agent generates its keypair locally and sends a CSR (correct — private key never leaves the device) or the server generates and ships the key (wrong — server holds the device identity), where the private key is stored on a SYSTEM-readable disk, and what the fingerprint is computed over (hardware IDs? cert thumbprint? a random device GUID?). A fingerprint derived from spoofable hardware identifiers undermines device-based licensing and anti-cloning; a server-generated private key undermines device identity. These choices are baked into the enrollment handshake built in Sprint 1.
- **Impact:** If the device private key is server-generated or stored unprotected, device identity can be cloned/impersonated, breaking device-based licensing (a core monetization mechanism) and any future mTLS. A weak/spoofable fingerprint lets one license be reused across many machines or lets an attacker masquerade as an enrolled device.
- **Recommendation:** Specify in Sprint 1: agent generates its keypair locally and submits a CSR during enrollment; the SaaS signs and returns a cert; the private key is stored with OS-level protection (Windows DPAPI/CNG machine-scope or, where available, TPM/CryptoAPI; on Linux root-only 0600 + future TPM). Define the fingerprint as the cert's SPKI thumbprint plus a persisted random device GUID (not raw hardware serials), and have the server bind the device record to that identity at enrollment. Lock this into the enrollment protocol now.

### SEC-6 — Config.yaml at C:\ProgramData\BeyzBackup holds enrollment/credentials with no specified at-rest protection or ACLs, contradicting 'never store secrets in source'
`High severity · High likelihood · **S1**`

- **Risk:** CLAUDE.md forbids secrets in source and mandates env vars, but the actual runtime stores config at C:\ProgramData\BeyzBackup\config.yaml (PRD/CLAUDE.md) — a directory that by default grants Users read access on Windows. The docs never state ACL hardening, encryption, or DPAPI protection for the enrollment token, agent certificate private key, storage credentials (SMB/SFTP/S3 secrets will land here in later sprints), or any wrapped key material from SEC-1. 'Use environment variables for sensitive configuration' is impractical for a Windows Service whose secrets must persist across reboots, so secrets will inevitably hit this file. A non-admin local user reading config.yaml is a direct credential-disclosure path.
- **Impact:** Any local low-privileged user or malware can read the enrollment token, device credentials, and (later) storage backend secrets straight off disk, enabling device impersonation, lateral movement to the customer's storage, and tenant compromise — with no exploit required, just a file read.
- **Recommendation:** In Sprint 1 define the on-disk secret model: lock C:\ProgramData\BeyzBackup ACLs to SYSTEM + Administrators only (remove Users), store secrets encrypted at rest (Windows DPAPI machine scope / Linux root-only 0600 with future keyring/TPM), and keep only non-sensitive settings in plaintext YAML. Have the installer set the restrictive ACLs at create-folder time (PRD req 15). Add a config schema field separating secret material so it is never logged.

### SEC-7 — Audit log / tamper detection are listed as features but have no integrity design — and the SYSTEM agent can rewrite its own logs
`Medium severity · Medium likelihood · **S1**`

- **Risk:** PRD lists 'Audit log' and 'Tamper detection'; SECURITY.md requires logging of enrollment, update, restore, failed integrity check, and auth failure; ROADMAP defers 'Audit Logging' to Sprint 8. Yet no integrity mechanism is specified: logs written by a SYSTEM-level process to C:\ProgramData\BeyzBackup\logs\ can be edited or deleted by that same process or by any attacker who reaches SYSTEM (which the agent itself runs as). With no append-only/hash-chaining/remote-shipping design, the audit trail is unreliable precisely in the ransomware/insider scenarios the product targets (ROADMAP Sprint 10 'Ransomware Detection'). The log format and event schema are decided in Sprint 1 ('structured logging,' PRD req 6).
- **Impact:** After a compromise (ransomware, malicious insider, or attacker pivoted to SYSTEM), the audit trail can be silently altered or wiped, destroying forensic evidence and breaking any compliance/SLA claim that depends on a trustworthy log — undermining the security value proposition.
- **Recommendation:** Design the Sprint 1 structured-log schema with security events as first-class records and a forward integrity mechanism: per-record hash chaining (each entry includes the hash of the prior) and near-real-time shipping of security events to the SaaS so the authoritative copy is off the endpoint and out of the agent's own reach. Plan immutable/WORM server-side storage for these events. Restrict log-dir ACLs (see SEC-6). Decide the event schema now so Sprint 8 only adds transport, not a breaking format change.

### SEC-8 — No documented control-protocol versioning or downgrade protection for enrollment/heartbeat/update — security upgrades (mTLS, signed updates) cannot be enforced later
`Medium severity · Medium likelihood · **S1**`

- **Risk:** The roadmap explicitly plans to strengthen security over time (mTLS in Sprint 8, signed-update hardening in Sprint 8), meaning Sprint 1 agents speak a weaker protocol than future agents/servers. Nothing in the docs specifies a protocol/version field or a minimum-security-version negotiation for the enrollment, heartbeat, task-polling, or update-manifest exchanges. Without an explicit version and a server-enforced floor, an attacker can force-downgrade a v2 (mTLS, pinned) agent/server to v1 (plain TLS, no pin) behavior, neutralizing the very hardening Sprint 8 adds. Adding a version field after thousands of agents are deployed requires backward-compat shims forever.
- **Impact:** Future security hardening becomes opt-in and bypassable: a network attacker downgrades agents to the weak Sprint 1 protocol, and the server cannot reject legacy/forged clients because no version floor was designed in. This silently caps the product's security at Sprint 1 levels for the install base.
- **Recommendation:** Add an explicit protocol_version (and min_supported_version) to every control-plane message and to the update manifest in Sprint 1, and have the server reject versions below a configurable floor. Define how the floor is raised (e.g., after agents auto-update) so mTLS and signed-update enforcement in Sprint 8 can be made mandatory rather than negotiable. This is a one-line-now / impossible-later decision.

### SEC-9 — Updater rollback + binary-replace flow trusts local files and lacks anti-rollback/version-floor and atomicity guarantees, enabling downgrade and TOCTOU attacks
`Medium severity · Medium likelihood · **S1**`

- **Risk:** PRD req 12 defines the updater skeleton: manifest check, signature verify, SHA256, stop service, replace binary, restart, rollback. The design verifies the new binary but says nothing about (a) preventing a signed-but-old (downgrade) binary from being installed to reintroduce a patched vulnerability — there is no minimum-version/anti-rollback check; (b) the window between signature/SHA verification and execution where the file on disk could be swapped (TOCTOU) if the staging directory is writable by non-admins; (c) what the rollback restores from and whether the rollback copy itself is integrity-checked. Because the updater runs as SYSTEM and replaces the SYSTEM-level agent, any flaw here is a privilege/persistence vector.
- **Impact:** An attacker (or a stale CDN/mirror) can push a validly-signed older agent with a known vulnerability and the updater installs it; or a non-admin-writable staging path lets a local user swap the verified binary before launch, achieving SYSTEM code execution. A rollback to an unverified backup copy can reintroduce a compromised binary.
- **Recommendation:** In the Sprint 1 updater design: enforce anti-rollback (reject manifests/binaries with version < currently installed unless an explicit signed downgrade flag is present), stage downloads only in a SYSTEM/Admin-only directory and re-verify the SHA256/signature of the exact bytes immediately before execution (verify-then-exec on the same locked handle to close TOCTOU), and integrity-check the rollback copy before restoring it. Pin the update-manifest source to the pinned SaaS endpoint (SEC-4) so the channel cannot be spoofed.

## 3. Scalability Risks

*The design is a pure pull/poll architecture (ARCHITECTURE.md: Enrollment → Heartbeat → Task Polling) with no push channel, no specified polling interval, no jitter, and no protocol versioning. At the stated target of thousands-to-tens-of-thousands of agents, this pattern creates predictable thundering-herd spikes, severe PostgreSQL write amplification from heartbeats, and connection-establishment load amplified by the future mTLS requirement. "Multi-tenant first" is asserted (CLAUDE.md) but no tenant partitioning, Redis queue topology, or per-tenant fairness strategy is defined. The deferred chunk/dedup/reference-counting subsystem (ROADMAP Sprint 5) has unbounded metadata growth that is not designed for, and zero-knowledge client-side encryption (SECURITY.md) silently eliminates cross-tenant storage dedup, which dominates the cost model. Several of these are foundational protocol/schema decisions that are cheap now and expensive to retrofit, so they are Sprint-1 relevant even though the heavy features ship later. The single most urgent Sprint-1 action is to nail down the agent↔SaaS wire protocol: interval+jitter config, schema/version envelope, monotonic state cursors, and a separation of heartbeat (presence) from task-fetch (work).*

### SCALE-1 — Unspecified, un-jittered polling interval guarantees thundering-herd spikes
`High severity · High likelihood · **S1**`

- **Risk:** PRD req 9-10 and ROADMAP Sprint 1 define a 'heartbeat sender' and 'task polling placeholder' but NO document specifies a polling interval, jitter, or backoff. Naive implementations use a fixed interval (e.g. every 30s/60s on a round number). With tens of thousands of agents that all (a) install during a business-hours MSP rollout, (b) reboot after Windows Patch Tuesday, or (c) reconnect after a SaaS deploy, their clocks align and they poll in synchronized waves. Aligned polling converts steady-state load into periodic spikes many multiples of the average.
- **Impact:** Synchronized request storms at fixed boundaries (e.g. :00 of every minute) saturate the API tier and PostgreSQL connection pool, causing timeouts and cascading retries that worsen the spike. A SaaS restart triggers a full reconnect stampede that can prevent the service from coming back up at all.
- **Recommendation:** In Sprint 1 make the heartbeat/poll interval a server-controllable config value returned in the heartbeat/enroll response (not hardcoded in the agent), and add mandatory randomized jitter (e.g. interval ± 20%) plus exponential backoff with full jitter on errors. Implement decorrelated jitter on reconnect after the SaaS is unreachable. Bake the 'server tells agent its next poll delay' field into the wire protocol now so cadence can be tuned centrally without shipping a new agent.

### SCALE-2 — Heartbeat-as-DB-write causes PostgreSQL write amplification and table bloat
`High severity · High likelihood · **S1**`

- **Risk:** The obvious implementation of 'heartbeat sender' (PRD req 9) updates a last_seen/status row in PostgreSQL on every heartbeat. At 20,000 agents heartbeating every 30s that is ~667 writes/sec of pure presence churn, each generating a new row version (PostgreSQL MVCC) and WAL traffic, hammering autovacuum and bloating the agents table and its indexes. This is write traffic that carries almost no business value but competes with real backup/manifest writes for IOPS and connections.
- **Impact:** Index/table bloat degrades query latency across the whole agents table (including the dashboard and licensing checks), autovacuum falls behind, WAL volume inflates replication and backup cost, and the primary DB becomes the scaling bottleneck long before backup throughput does.
- **Recommendation:** Do not write heartbeats to PostgreSQL on the hot path. In Sprint 1, define presence state to live in Redis (e.g. a key per agent with TTL, or a sorted set keyed by last-seen timestamp) and only flush state transitions (online↔offline) to PostgreSQL. Design the heartbeat payload now to be minimal (agent fingerprint + version + status enum) and explicitly separate it from the heavier task-poll response so presence updates never touch the relational store.

### SCALE-4 — "Multi-tenant first" is asserted but no tenant partitioning or data-isolation-at-scale strategy is defined
`High severity · Medium likelihood · **S1**`

- **Risk:** CLAUDE.md states 'Multi-tenant first' and ARCHITECTURE.md lists Tenant Management, but no document specifies HOW tenants are partitioned: shared schema with tenant_id columns, schema-per-tenant, or DB-per-tenant; whether tenant_id is part of every primary key and index; or how a single huge MSP tenant (managing thousands of devices) is isolated from noisy-neighbor effects. The agent fingerprint and enrollment identifiers minted in Sprint 1 implicitly fix the tenancy key scheme.
- **Impact:** Retrofitting a tenant_id into primary keys, indexes, foreign keys, and the manifest/metadata schema after data exists is a massive, risky migration. Without it, queries scan across all tenants, a large MSP can starve smaller tenants, and per-tenant data residency/deletion (a likely SMB/EU requirement) becomes impractical. Cross-tenant data leakage risk rises if isolation is bolted on later.
- **Recommendation:** Decide the partitioning model in Sprint 1 and encode tenant_id into the enrollment/fingerprint identity and into every entity's key from day one (recommend shared-schema with mandatory tenant_id as the leading column of composite primary keys and indexes, plus row-level security or a query-layer tenant guard). Reserve the option of PostgreSQL declarative partitioning by tenant_id for hot tables (agents, heartbeats-flush, manifests). Document the noisy-neighbor / large-tenant sharding escape hatch even if not yet built.

### SCALE-6 — Dedup index and chunk reference-counting metadata grow unbounded with no scaling design
`High severity · Medium likelihood · **S1**`

- **Risk:** PRD (lines 50-52) and ROADMAP Sprint 5 commit to chunk-based backup with reference counting, and ARCHITECTURE.md's data flow (Chunking → Hashing → ... → Manifest) implies a chunk index. A global chunk catalog plus per-chunk reference counts is the highest-cardinality dataset in the entire system — potentially billions of rows across the fleet. No document addresses index sharding, the hot-row contention of ref-count increments/decrements during concurrent backups/deletes, or how dedup scope (per-device / per-tenant / global) is bounded. The manifest format and chunk-hash identity decided in early sprints lock this in.
- **Impact:** A monolithic chunk/ref-count table becomes the system's scaling and contention bottleneck: ref-count updates create lock contention and bloat under concurrent retention/GC, the dedup lookup hot path slows as the index grows, and orphaned-chunk garbage collection becomes an expensive full-scan. Getting ref-counting wrong risks deleting still-referenced chunks — silent restore corruption, the worst outcome for a backup product.
- **Recommendation:** Even though it ships in Sprint 5, decide the chunk-identity and dedup-scope model NOW, because the agent fingerprint, chunk hash (BLAKE3/SHA256), and manifest schema are Sprint-1/early decisions. Choose a bounded dedup scope (recommend per-tenant, not global — also required by zero-knowledge encryption per SCALE-8), plan to partition the chunk index by tenant_id, and design ref-counting as an append-only event log reconciled by a background GC rather than synchronous in-place counter updates to avoid hot-row contention. Specify content-addressed chunk IDs so the index is shardable by hash prefix.

### SCALE-3 — Pull-only model wastes the majority of requests and scales work inversely to its value
`Medium severity · Medium likelihood · **S1**`

- **Risk:** ARCHITECTURE.md lists the agent loop as Heartbeat + Task Polling with no server-initiated push, and SECURITY.md lists mTLS only as a 'Future' feature. In a backup product most agents are idle most of the time (one nightly job), so the overwhelming majority of poll requests return 'no work.' To keep job-start latency low you must poll frequently, which multiplies empty requests; to reduce load you lengthen the interval, which delays job dispatch and ad-hoc restores. This is a structural inefficiency: request volume grows with fleet size while useful dispatched work does not.
- **Impact:** Either high infrastructure cost for mostly-empty polls, or sluggish, unpredictable job/restore start times that hurt the product's core promise. The mTLS handshake (a future requirement) makes each connection far more expensive, so the empty-poll cost compounds once security hardening lands in Sprint 8.
- **Recommendation:** Decide the long-lived connection strategy NOW even if implemented later: design the protocol so the heartbeat response can carry a 'work available' flag and a 'long-poll/hold for N seconds' hint, enabling a cheap migration to long-polling or a push channel (WebSocket/SSE/gRPC stream) without a wire-protocol break. Ensure HTTP keep-alive / connection reuse is mandated for the agent client in Sprint 1 so the future mTLS handshake is amortized across many requests rather than paid per poll.

### SCALE-5 — Redis named as the queue with no topology, durability, or per-tenant fairness design
`Medium severity · Medium likelihood`

- **Risk:** CLAUDE.md designates Redis as 'Queue' with zero elaboration anywhere. Open questions that determine whether it scales: is it a simple LIST/PUBSUB, Redis Streams with consumer groups, or just a cache? How is task durability guaranteed if Redis restarts (default Redis can lose data)? Is there per-tenant queue isolation so one MSP's bulk job enqueue cannot delay every other tenant's tasks? Single-instance Redis is also a vertical-scaling ceiling and a single point of failure for task dispatch.
- **Impact:** A non-durable queue silently drops backup/restore tasks on failover (violates the product's 'restore reliability over speed' principle). Lack of per-tenant fairness lets one large tenant's burst monopolize workers (head-of-line blocking). A single Redis becomes a throughput and availability bottleneck for the whole fleet.
- **Recommendation:** Specify the queue design before task dispatch is built (Sprint 2-3): use Redis Streams with consumer groups (or an alternative durable broker) with AOF persistence enabled, define per-tenant stream/partition keys to enforce fairness, set a Redis Cluster / Sentinel HA topology, and define idempotent, at-least-once task semantics with a dedup key so redelivery is safe. Make the task identifier and idempotency key part of the Sprint-1 task-poll protocol placeholder.

### SCALE-7 — Monitoring/Reporting queries run against the live OLTP store with no read-scaling plan
`Medium severity · Medium likelihood`

- **Risk:** ARCHITECTURE.md gives the SaaS Platform 'Monitoring' and 'Reporting' responsibilities, and ROADMAP Sprint 9 adds Reports and Storage Warnings. Dashboards that aggregate per-tenant device status, last-backup times, storage-quota usage, and job history across tens of thousands of agents are heavy analytical queries. If they hit the same PostgreSQL primary that is already absorbing heartbeat flushes (SCALE-2) and manifest writes, a single MSP opening a 'fleet status' page can run fleet-wide aggregations that stall the write path.
- **Impact:** Reporting queries contend with operational writes, degrading both dashboard responsiveness and backup/heartbeat ingestion. Storage-quota and licensing checks (which gate backups) slow down. As tenant count grows, dashboards time out, the most visible symptom of scaling failure to paying MSP customers.
- **Recommendation:** Plan read/write separation early: route reporting and dashboard reads to a read replica, and pre-aggregate per-tenant rollups (last-backup status, storage used, device counts) into summary tables or a materialized view refreshed on a schedule rather than computing them live. In Sprint 1, ensure the event/metric data needed for these rollups (job outcomes, heartbeat transitions, bytes stored) is captured in an append-friendly, time-series-shaped form so it can later feed a separate analytics path without schema rework.

### SCALE-8 — Zero-knowledge client-side encryption eliminates cross-tenant dedup, inflating the storage cost model
`Medium severity · Medium likelihood · **S1**`

- **Risk:** SECURITY.md mandates client-side encryption where 'Server must never access plaintext files,' and the product is sold on customer-side zero-knowledge. With AES-256-GCM applied per-agent before upload, identical plaintext chunks encrypt to different ciphertext across tenants (and across keys), so the server cannot dedup across customers — exactly the savings that makes backup storage economics work for Veeam/Acronis-class products. No document reconciles the dedup feature (PRD/ROADMAP Sprint 5) with the encryption requirement, nor states whether convergent/content-derived encryption is in scope.
- **Impact:** Storage cost on Beyz Cloud is far higher than a naive dedup model would predict, because dedup is limited to within a single key scope. If the business model or pricing was built assuming Acronis-like global dedup ratios, gross margins on the cloud-storage tier are overstated. Conversely, if convergent encryption is later introduced to recover dedup, it weakens the zero-knowledge guarantee (confirmation/known-plaintext attacks) — a security regression.
- **Recommendation:** Resolve the encryption-vs-dedup tension explicitly before pricing and the chunk schema are locked. Recommend: scope dedup to per-tenant (or per-encryption-key) only, derive chunk encryption from a tenant-held key so chunks dedup within a tenant but not across tenants, and model the cloud-storage cost using realistic intra-tenant dedup ratios (not cross-customer). Document this as a deliberate security/cost tradeoff so the licensing/storage-quota pricing (a Sprint-1-adjacent business decision) is built on correct assumptions.

### SCALE-9 — No protocol versioning or schema envelope means fleet-wide upgrades become a scaling hazard
`Medium severity · Medium likelihood · **S1**`

- **Risk:** Sprint 1 defines the enrollment, heartbeat, and task-poll wire formats (PRD req 7-10) and an auto-updater (PRD req 11-12), but no document mentions an API/protocol version field or a schema-evolution strategy. With tens of thousands of self-updating agents, you will permanently have a mix of agent versions in the field. Without an explicit version negotiated on every request, any change to the heartbeat/task payload forces a flag-day, and a bad auto-update pushed fleet-wide (the updater is built Sprint 1) can cause a synchronized reconnect/re-enroll storm — a scaling event, not just a correctness event.
- **Impact:** Inability to evolve the protocol without breaking older agents; risky big-bang migrations; and an auto-update that, if it triggers mass re-enrollment or tight-loop retries, becomes a self-inflicted DDoS on the SaaS. Version skew also complicates the server's ability to throttle or stage rollouts.
- **Recommendation:** Add an explicit protocol/schema version field and a forward-compatible envelope (ignore-unknown-fields) to every agent↔SaaS message in Sprint 1. Build staged/canary rollout capability into the updater (server controls what fraction of the fleet updates) and ensure update-induced reconnects inherit the SCALE-1 jitter/backoff so a fleet-wide update cannot stampede. Make the server able to reject or gracefully degrade unknown versions rather than erroring in a retry loop.

## 4. Backup-Engine Design Risks

*The backup engine is the heart of this product, yet the design documents describe it only as a single linear pipeline (ARCHITECTURE.md: Files->Chunking->Hashing->Compression->Encryption->Upload->Manifest->Restore Point) with one-line roadmap goals ("Chunk Engine", "Reference Counting", "Manifest"). The most consequential and hardest-to-reverse design decisions are therefore undecided and undocumented: the chunking strategy (fixed vs content-defined), the dedup-vs-encryption tension (the documented order encrypt-after-compress destroys cross-file/cross-tenant dedup, and the only fix — convergent encryption — leaks plaintext equality and breaks the zero-knowledge claim in SECURITY.md), the hash function semantics, the manifest format and its versioning, crash/idempotency semantics, and reference-counting/garbage-collection safety. Although the engine ships in Sprints 3-6, the manifest format, chunk identity scheme, hashing choice, key-derivation model, and protocol versioning must be decided NOW (Sprint 1) because they are foundational to the agent's config, enrollment-derived keying, and on-wire/on-disk formats — retrofitting them after agents are deployed in the field is extremely expensive. Most risks below are Sprint 1-relevant for that reason. The dominant theme: the product makes strong promises (zero-knowledge encryption, dedup, restore correctness "higher priority than backup speed") that are in direct, undocumented tension with each other, and nobody has written down how the tradeoff is resolved.*

### BKP-1 — Encryption-after-compression destroys cross-file/cross-tenant dedup; the only fix (convergent encryption) breaks the zero-knowledge guarantee
`Critical severity · High likelihood · **S1**`

- **Risk:** ARCHITECTURE.md fixes the order as Chunking -> Hashing -> Compression -> Encryption -> Upload. AES-256-GCM (CLAUDE.md) uses a random nonce per encryption, so two identical plaintext chunks produce different ciphertexts. That means dedup can only ever work on data that is identical AND encrypted with the same key+nonce — effectively no cross-file or cross-tenant dedup at all. The PRD sells 'chunk-based backup' and ROADMAP Sprint 5 promises 'Reference Counting' (which only pays off with real dedup), but with the documented pipeline the dedup index is built over hashes of plaintext (the 'Hashing' step is before encryption) while storage holds non-deduplicable ciphertext — a contradiction. The classic resolution is convergent encryption (derive the key/nonce from the plaintext hash), but that makes identical plaintext produce identical ciphertext, which leaks plaintext equality and enables confirmation-of-file and chosen-plaintext attacks — directly violating SECURITY.md 'Server must never access plaintext files' and the zero-knowledge / 'Storage compromise must not expose customer data' promises. No document acknowledges this tension or picks a side.
- **Impact:** If unaddressed, the team will discover mid-Sprint-5 that reference counting and dedup deliver almost no storage savings (undermining the storage-quota licensing economics and the 'lightweight' positioning), OR an engineer will silently introduce convergent encryption to make dedup 'work' and quietly break the zero-knowledge security guarantee that is the product's core differentiator and a contractual promise to SMB/MSP customers. Either outcome is a strategic failure discovered late.
- **Recommendation:** Decide and document the dedup scope NOW: (a) per-agent (per-device) dedup only, using a per-device data-encryption key, so identical chunks within one device dedup but nothing leaks across tenants — this preserves zero-knowledge and still gives strong incremental savings; explicitly forbid cross-tenant dedup in the design. (b) Make the manifest/chunk-ID scheme store the plaintext-hash as the dedup key but the storage object key as a keyed value (HMAC(device_key, plaintext_hash)) so the storage backend never sees a global content hash. Write this into SECURITY.md and the manifest spec in Sprint 1 because the chunk-ID and key-derivation model is baked into config/enrollment keying.

### BKP-3 — Manifest format and its versioning are completely unspecified — the manifest is the single point of failure for every restore
`Critical severity · High likelihood · **S1**`

- **Risk:** ARCHITECTURE.md ends the pipeline at 'Manifest Creation -> Restore Point' and ROADMAP Sprint 3 lists 'Manifest' as a one-word goal. There is no spec for what the manifest contains, its serialization format, its schema version, how it references chunks, how it is encrypted/signed, or where it is stored. The manifest is the only thing that maps a restore point to its constituent chunks; if it is lost, corrupt, or unversioned, every backup it describes is unrecoverable — which directly contradicts CLAUDE.md 'Restore reliability has higher priority than backup speed' and 'Backup integrity first'. A manifest without an explicit schema-version field cannot evolve once V1 agents are in the field, and a manifest stored only locally on the agent is lost when the device dies (the exact disaster-recovery case backups exist for).
- **Impact:** A missing version field forces a hard format freeze or a risky in-place migration of every customer's restore points. A manifest that isn't redundantly stored alongside the chunks means a dead/ransomwared device cannot be restored at all. A manifest that isn't integrity-protected lets storage tampering silently corrupt restores. Any of these turns a 'successful backup' into an unrestorable one — the worst possible failure for a backup product.
- **Recommendation:** Specify the manifest in Sprint 1 design: (1) an explicit 'manifest_schema_version' integer; (2) per-restore-point metadata (device fingerprint, timestamp, parent/base restore-point id for incrementals, chunker params, hash algo id, crypto algo id); (3) an ordered file->chunk-list mapping with each chunk's plaintext hash, ciphertext storage key, size, and compression flag; (4) the manifest itself encrypted with the device key and integrity-tagged; (5) the manifest uploaded to the storage backend (not only kept locally) and its own hash recorded server-side so restore can verify it. Pick a stable serialization (e.g., canonical JSON or protobuf) and freeze it.

### BKP-8 — Per-device data-encryption-key derivation and custody are undefined, yet enrollment (Sprint 1) is where keying must originate to keep dedup and zero-knowledge coherent
`Critical severity · Medium likelihood · **S1**`

- **Risk:** Key Management is listed only in Sprint 4 (ROADMAP), and SECURITY.md lists 'Hardware-backed keys' as a future feature, but the backup engine's correctness depends on the keying model being fixed at enrollment in Sprint 1. CLAUDE.md mandates client-side/zero-knowledge encryption and forbids storing keys in source/server, while BKP-1's dedup resolution requires a stable per-device data key to compute deterministic chunk IDs. If the data-encryption key is generated lazily in Sprint 4 with no link to the Sprint 1 enrollment/agent-certificate flow, the team faces an impossible choice: either the server can recover the key (breaking zero-knowledge) or there is no key-escrow and a user who loses the device/passphrase permanently loses all backups (a support and liability nightmare for SMB/MSP customers who expect the vendor to help them restore).
- **Impact:** Retrofitting the keying model after agents are enrolled means re-keying and re-encrypting every customer's data — effectively impossible at fleet scale. Getting it wrong yields either a broken zero-knowledge promise (security/contractual breach) or unrecoverable customer backups (the product failing at its one job).
- **Recommendation:** Define the key hierarchy in Sprint 1: derive (or wrap) a per-device data-encryption key at enrollment, bound to the agent certificate generated post-enrollment (SECURITY.md), with an explicit, documented decision on recovery — e.g., a customer-controlled recovery key / escrow wrapped with a customer passphrase that the server stores only in wrapped form (preserving zero-knowledge while enabling recovery). Record the key-id and KDF parameters in the manifest so chunk IDs (BKP-1) and decryption are reproducible. Write this into SECURITY.md before any crypto code ships.

### BKP-2 — Chunking strategy (fixed-size vs content-defined) is undecided — fixed-size chunking destroys incremental dedup on file edits/inserts
`High severity · High likelihood · **S1**`

- **Risk:** No document specifies the chunking algorithm. ARCHITECTURE.md says only 'Chunking'; ROADMAP Sprint 5 says only 'Chunk Engine'. Fixed-size chunking is trivial to implement and the path of least resistance, but it has a catastrophic failure mode: inserting or deleting even one byte near the start of a file shifts every subsequent chunk boundary, so a tiny edit re-uploads the entire file (the 'boundary shift' problem). Content-defined chunking (CDC) via a rolling hash (Rabin/buzhash/FastCDC) anchors boundaries to content so edits only re-chunk the local region. For a backup product whose value is efficient incrementals (Sprint 5) over documents, VMs, mailboxes, and databases that get edited in place, fixed-size chunking would make incrementals nearly as large as fulls.
- **Impact:** Without CDC, incremental backups balloon, upload bandwidth and customer storage quotas blow out, and the product loses its efficiency advantage over Veeam/Acronis. If chunk size and algorithm are chosen ad hoc in Sprint 5 and agents are already deployed, changing the chunking parameters re-chunks everyone's data and breaks dedup against existing restore points.
- **Recommendation:** Commit to content-defined chunking (FastCDC is the modern, fast choice) with documented min/avg/max chunk sizes (e.g., 2/8/16 MiB or 4/16/64 KiB depending on dedup target), and record the chunker version + parameters as fields in the manifest so chunking can be migrated later without ambiguity. Decide the parameter set in Sprint 1 design even though code ships in Sprint 5, because the manifest format and chunk-ID scheme depend on it.

### BKP-4 — No crash-consistency / idempotency / resume-integrity model — interrupted backups risk corrupt restore points and orphaned or double-counted chunks
`High severity · High likelihood`

- **Risk:** PRD lists 'Resumable upload' and 'Chunk upload' and Sprint 5 adds 'Reference Counting', but no document describes the consistency model when a backup is interrupted (crash, network loss, service stop — and note the updater in Sprint 1 explicitly stops/replaces/restarts the agent service, which can interrupt a running job). Key unanswered questions: Is a restore point committed atomically only after ALL chunks + manifest are durably uploaded? On resume, how does the agent know which chunks already landed without re-uploading or, worse, double-incrementing a reference count? If reference counting increments as chunks upload and the backup then fails, do orphaned refs leak storage forever or does GC delete chunks a half-written manifest still needs? Resumable upload without per-chunk integrity verification on the server side means a truncated/corrupt chunk can be marked 'done'.
- **Impact:** Partially-completed backups that present as valid restore points but fail on restore; reference-count drift that either leaks storage (cost/quota inflation) or — far worse — lets garbage collection delete live chunks, silently corrupting other customers' restore points that shared them. This is the canonical way dedup+refcount backup systems lose data.
- **Recommendation:** Define the commit protocol now: upload all chunks first (content-addressed, idempotent PUTs keyed by hash so re-uploading is a safe no-op), verify each chunk's hash server-side on receipt, then write the manifest LAST as the atomic commit point; a restore point is valid iff its manifest exists and references only chunks the server has confirmed. Make reference counting derive from committed manifests (mark-and-sweep GC over manifests) rather than incrementing during upload, so an aborted backup leaves only unreferenced chunks that GC reclaims safely. Add a grace period / generation marker before GC deletes any chunk to avoid races with in-flight backups.

### BKP-5 — BLAKE3-vs-SHA256 left as 'preferred/acceptable' with no committed on-disk algorithm identifier — a later switch breaks dedup and integrity verification
`High severity · Medium likelihood · **S1**`

- **Risk:** CLAUDE.md and PRD leave the hash undecided: 'BLAKE3 preferred, SHA256 acceptable' / 'SHA256 or BLAKE3'. The hash is doing double duty: chunk identity for dedup AND integrity verification before restore (SECURITY.md 'Integrity verification required before restore'). These are different requirements — chunk identity wants speed and a wide tree (BLAKE3 excels), while SECURITY.md separately mandates 'SHA256' for the signed update path (different subsystem). If the engine ships with one hash and later 'upgrades' to the other, every existing chunk ID changes, dedup against historical data breaks, and old manifests become unverifiable unless each chunk record carries an explicit algorithm tag. Treating 'SHA256 acceptable' as license to mix the two without a discriminator is the trap.
- **Impact:** An ad hoc or inconsistent hash choice across modules means restore points created under one algorithm cannot be deduped or integrity-checked after a switch, and a security reviewer cannot reason about collision resistance for the integrity guarantee. Silent truncation or mismatched digest lengths between modules causes verification false-positives.
- **Recommendation:** Commit to BLAKE3 as the single chunk-identity-and-integrity hash for the backup engine (it is faster and keyed-mode supports the per-device keyed HMAC-style chunk IDs from BKP-1), keep SHA256 strictly for the update-signature path where SECURITY.md mandates it, and store an explicit 'hash_algo' identifier in every manifest and chunk record so the format is future-proof. Document the full 256-bit digest length and forbid truncation. Decide in Sprint 1 because the chunk-ID format is foundational.

### BKP-6 — Open-file / locked-file / VSS handling deferred to Sprint 10, but the agent will silently produce inconsistent or skipped backups starting in Sprint 3
`High severity · High likelihood · **S1**`

- **Risk:** ROADMAP defers VSS to Sprint 10, yet Full Backup ships in Sprint 3 and runs as a Windows Service (CLAUDE.md/PRD). Without VSS, on Windows the agent cannot read files that applications hold open with exclusive locks (Outlook PST/OST, SQL/MDF, registry hives, any open Office doc), and files that change during the multi-minute backup are captured torn (half-old/half-new bytes). The pipeline in ARCHITECTURE.md chunks-then-hashes a file as it reads it; with no snapshot, a file modified between the start and end of chunking yields a chunk set that never existed as a coherent file state — a corrupt restore that still 'verifies' because its own internal hashes are self-consistent. The product targets hotels/SMBs/MSPs whose most valuable data (mail stores, databases) is exactly the always-open kind.
- **Impact:** Customers run 'successful' backups for months, then on restore discover their PST/database/VM is corrupt or was silently skipped — the textbook way backup products lose customer trust and face liability. 'Restore correctness first' (CLAUDE.md) is violated in practice from Sprint 3 onward despite VSS being scheduled for Sprint 10.
- **Recommendation:** Do NOT ship file-by-file reads without consistency. Pull a minimal VSS snapshot capability forward (or at minimum a documented snapshot abstraction) to before Sprint 3's full backup, OR explicitly gate Sprint 3 to back up only closed/static files and have the agent detect-and-skip-with-loud-warning any locked or actively-changing file (record skipped files in the manifest and surface them to the SaaS panel). Add a per-file change-detection check (mtime/size before vs after read) that fails the file's chunk set rather than committing a torn copy. Decide the consistency contract in Sprint 1 design.

### BKP-7 — No backup-format / protocol version negotiation between agent and SaaS — auto-update will field a fleet of mixed-version agents writing incompatible data
`High severity · Medium likelihood · **S1**`

- **Risk:** The product has agent auto-update (PRD #8, updater in Sprint 1) and is multi-tenant SaaS, so at any moment the field will run many agent versions simultaneously. Nothing in the docs defines a version-negotiation handshake for the backup wire/storage format, the manifest schema, the chunker params, or the crypto envelope. Sprint 1 builds enrollment, heartbeat, and task-polling — the exact place a 'backup_format_version' / 'capabilities' field belongs — but it is not mentioned. If Sprint 5 changes chunking or Sprint 4 changes the crypto envelope, a newer agent can produce restore points an older restore path (or a peer agent doing dedup against shared chunks) cannot read, and vice versa.
- **Impact:** Auto-update — a core selling point — becomes a data-integrity hazard: an update silently changes the format and either orphans old restore points or makes new backups unrestorable by tooling that expects the old format. Mixed-fleet incompatibility is discovered only after thousands of agents have written incompatible data.
- **Recommendation:** Add capability/version negotiation to the Sprint 1 enrollment + heartbeat protocol: the agent reports its 'agent_version' and a set of 'supported_format_versions' (manifest, chunk, crypto-envelope), and the SaaS records and can refuse/flag incompatible combinations. Embed the same format-version integers in the manifest (see BKP-3). This is cheap to add now and impossible to retrofit cleanly once the fleet is deployed.

## 5. Restore-Engine Design Risks

*The restore path is the product's stated top priority ("Restore correctness first", "Restore reliability > backup speed"), yet it is the least-specified part of the design. SECURITY.md devotes exactly two sentences to restore ("Integrity verification required before restore. Corrupted backups must be quarantined.") with no mechanism, scope, or failure model. The roadmap sequences Key Management (Sprint 4) and Restore (Sprint 6) AFTER real backups are produced (Sprint 3), so the on-disk manifest format, chunk-reference scheme, encryption-key custody, and hash-algorithm identifier are all created before anyone designs the restore that must consume them — guaranteeing expensive retrofits. The most severe restore risks are foundational and decided in Sprint 1: (1) zero-knowledge client-side encryption with no key-escrow/recovery design means a lost key permanently destroys restorability; (2) the manifest is the single point of restore failure but has no described integrity protection, signing, versioning, or redundancy; (3) "BLAKE3 preferred, SHA256 acceptable" without a recorded algorithm-id breaks cross-version integrity verification; (4) config/secret layout fixed in Sprint 1 reserves no fields for key-id, manifest-schema-version, or crypto-version. Several of these MUST be designed for now even though restore ships in Sprint 6, because they are cheap to specify today and prohibitively expensive to retrofit once production backups exist in customer storage.*

### RST-1 — Zero-knowledge encryption with no key-escrow or key-recovery design — lost key = permanently unrestorable backups
`Critical severity · High likelihood · **S1**`

- **Risk:** CLAUDE.md mandates client-side encryption and 'Never store encryption keys' / 'Server must never access plaintext' (SECURITY.md). The roadmap puts 'Key Management' in Sprint 4 and Restore in Sprint 6, but NO document describes where the data-encryption key lives, how it is derived/wrapped, or how it is recovered if the customer loses it. With pure zero-knowledge encryption, if the agent host is wiped (the exact bare-metal/disaster scenario this product sells against) and the key was only on that host, every backup in customer + Beyz Cloud storage becomes ciphertext garbage — restore is mathematically impossible.
- **Impact:** A hotel/MSP whose server dies restores from Beyz Backup, the original selling point, and gets nothing back because the only copy of the key died with the machine. Total, unrecoverable data loss for the customer and catastrophic reputational/legal liability for Beyz. This is the single most likely way the product fails at its one job.
- **Recommendation:** Decide the key hierarchy NOW (Sprint 1) even though crypto ships in Sprint 4: define a per-tenant/per-device data key wrapped by (a) a customer passphrase-derived KEK AND (b) an optional escrow KEK held by the SaaS in a way that preserves zero-knowledge (e.g. wrapped with a customer-controlled recovery public key, or Shamir-split with customer-held shares). Reserve a 'key_id' and 'key_wrap_version' field in config.yaml and in the enrollment/registration payload now. Document explicitly that a backup is unrestorable without the key and force a 'recovery key downloaded/confirmed' gate at enrollment. Without an escrow decision before the first real backup (Sprint 3), you can never retrofit recovery onto already-encrypted historical data.

### RST-2 — Manifest is the single point of restore failure but has no integrity protection, signing, redundancy, or schema versioning
`Critical severity · High likelihood · **S1**`

- **Risk:** ARCHITECTURE.md data flow ends 'Upload -> Manifest Creation -> Restore Point'. The manifest is what maps a logical file to its ordered chunk list / hashes / restore-point lineage — without it the chunks are unrecoverable even though they exist. Yet nothing describes the manifest being signed, integrity-hashed, stored redundantly, or versioned. SECURITY.md's 'integrity verification before restore' implicitly covers data chunks but never mentions verifying the manifest itself. The manifest is born in Sprint 3, long before the Sprint 6 restore engine that depends on it is designed.
- **Impact:** A single corrupted/truncated/tampered manifest in customer-owned SMB/SFTP storage renders an entire restore point or backup chain unrestorable while all chunk data is intact and present. An attacker who can write to storage can also silently rewrite chunk references to redirect or corrupt restores. With no schema version, the Sprint 6 restore engine cannot reliably parse manifests produced by Sprint 3-5 agents.
- **Recommendation:** Specify the manifest contract NOW: include a mandatory 'manifest_schema_version' integer, an embedded BLAKE3/SHA256 self-hash, and a detached signature (using the agent certificate generated at enrollment, which already exists in Sprint 1). Mandate redundant manifest storage (e.g. manifest copied to both customer storage and Beyz Cloud, or N copies). Make manifest integrity + signature verification the FIRST step of restore, before any chunk is read. Reserve the schema-version field in the registration/manifest format design that lands in Sprint 1 planning so Sprint 3 emits it from day one.

### RST-3 — 'BLAKE3 preferred, SHA256 acceptable' without a recorded per-object algorithm identifier breaks integrity verification at restore
`High severity · High likelihood · **S1**`

- **Risk:** CLAUDE.md states hashing is 'BLAKE3 preferred, SHA256 acceptable' — i.e. a per-build or per-agent choice. SECURITY.md requires 'Integrity verification required before restore'. If a chunk/manifest hash is stored as a bare hex string with no algorithm tag, a restoring agent cannot know whether a 64-hex-char value is SHA256 or a truncated BLAKE3, and a backup written by a BLAKE3 agent cannot be verified by a SHA256-only agent (and vice versa). Note also SECURITY.md's update path hardcodes 'SHA256', creating an inconsistency.
- **Impact:** Cross-version and cross-agent restore integrity checks fail or, worse, silently pass against the wrong algorithm, defeating the 'verify before restore' guarantee and the 'corrupted backups must be quarantined' control. A backup made today may be unverifiable by next year's agent, directly violating cross-version restore.
- **Recommendation:** Mandate an algorithm-tagged hash format everywhere NOW (e.g. 'blake3:<hex>' / 'sha256:<hex>') and store the chosen algorithm in the manifest header and config.yaml. Pick ONE default algorithm for v1 (recommend BLAKE3 with SHA256 as an explicitly-tagged fallback) rather than leaving it to per-build chance, and require the restore verifier to dispatch on the tag. Reconcile the updater's hardcoded SHA256 with this scheme. This is a one-line format decision now that is unfixable across historical backups later.

### RST-4 — No chunk reference-counting / chain-completeness model means incremental restores silently fail when a referenced chunk or parent restore point is missing
`High severity · Medium likelihood · **S1**`

- **Risk:** PRD lists 'Chunk-based backup', 'Incremental backup later', 'Reference counting later'; roadmap puts reference counting + retention in Sprint 5, restore in Sprint 6. Rebuilding a file at restore point N requires every chunk it references, which may live in any earlier restore point. Nothing defines how the restore engine validates that the FULL chain and ALL referenced chunks are present before starting, nor how retention/pruning (Sprint 5) is prevented from deleting a chunk still referenced by a retained restore point.
- **Impact:** Retention deletes a chunk that an older retained restore point still needs (dangling reference), or a partially-uploaded incremental leaves a gap; the user starts a restore that runs for hours and then fails or produces a corrupt file because chunk #4,217 is gone. This is the classic dedup-backup data-loss bug and it is invisible until someone actually restores.
- **Recommendation:** Define the chunk-reference and chain-completeness invariant before retention ships: every restore point manifest must enumerate its complete chunk dependency set, and reference counting (Sprint 5) must guarantee no chunk is pruned while any retained restore point references it. Add a mandatory pre-restore 'completeness check' that verifies every referenced chunk exists and hashes correctly BEFORE writing any output, failing fast with a precise missing-chunk list. Specify the manifest fields that carry parent-restore-point linkage now so Sprint 3/5 manifests are forward-compatible.

### RST-5 — Restore atomicity, partial-failure recovery, and resumability are completely unspecified
`High severity · Medium likelihood`

- **Risk:** Backup explicitly has 'Resumable upload' (PRD), but the symmetric restore path has no resumable-download, no atomicity, and no partial-failure recovery design anywhere. SECURITY.md mentions quarantine of corrupt backups but says nothing about what happens when a restore is interrupted (network drop, agent crash, disk full) mid-write — exactly the conditions present in a real disaster recovery.
- **Impact:** A restore that dies at 80% over a flaky SMB/SFTP link leaves half-written files in place, potentially overwriting the user's existing good data with truncated garbage and with no clean resume — turning a recoverable incident into permanent loss. Large multi-GB restores over WAN are precisely where interruptions are most likely.
- **Recommendation:** Adopt atomic restore semantics: restore each file to a temp path and atomically rename on success; never overwrite the target until the file is fully written and hash-verified. Make restore resumable with per-file/per-chunk progress checkpointing (mirror the resumable-upload design). Define explicit behavior for disk-full/permission-denied mid-restore. This is a Sprint 6 implementation, but the manifest must carry the per-file size/hash needed to verify completion — reserve those fields now.

### RST-6 — Cross-platform ACL / ownership / permission restoration is not modeled, and Windows<->Linux semantics don't map
`High severity · Medium likelihood · **S1**`

- **Risk:** The product targets both Windows and Linux (PRD), and bare-metal/device restore is a Sprint 6 goal. Restoring file CONTENT is the easy part; restoring Windows NTFS ACLs/SIDs/ownership/ADS and Linux POSIX mode/uid/gid/xattrs/symlinks is where backup products live or die. No document mentions capturing or restoring this metadata, and SIDs are machine/domain-specific so a restore to a rebuilt or different host can produce broken or insecure permissions.
- **Impact:** A bare-metal restore brings files back but with wrong ownership or wide-open/locked ACLs — applications fail to start, security is silently broken (e.g. confidential shares world-readable), or admins can't access their own data. For an SMB/hotel restoring after ransomware, broken permissions can make the restore unusable, defeating the product's core promise.
- **Recommendation:** Define a platform-tagged metadata model in the manifest now (capture NTFS security descriptor + ownership on Windows; mode/uid/gid/xattrs/symlink target on Linux) so Sprint 3 backups record it from the start — you cannot restore metadata you never captured. Specify SID/uid remapping behavior for restore-to-different-host, and an option to restore data-only vs data+permissions. Decide and document that cross-OS restore (Windows backup -> Linux target) does NOT carry ACLs, rather than discovering it in the field.

### RST-7 — 'Integrity verification' and 'quarantine' are asserted but undefined — no scope, no trigger, no quarantine mechanics, no restore audit log
`Medium severity · Medium likelihood · **S1**`

- **Risk:** SECURITY.md: 'Integrity verification required before restore. Corrupted backups must be quarantined.' and lists 'Restore' and 'Failed Integrity Check' as loggable security events. But there is no definition of WHAT is verified (chunk hashes? manifest signature? full chain?), WHEN (every restore vs scheduled scrub), what 'quarantine' physically means in customer-owned SMB/SFTP/S3 storage the agent may only have write access to, or what the restore audit record must contain. Audit logging itself is deferred to Sprint 8.
- **Impact:** Without a concrete spec, teams implement a token hash check that misses real corruption, and 'quarantine' is undefined for customer storage where the agent often cannot move/lock objects — so corrupt backups silently remain in the restore candidate set. Missing structured restore audit events make post-incident forensics (who restored what, did integrity pass) impossible, weakening the ransomware/compliance story.
- **Recommendation:** Write a concrete integrity spec: verify manifest signature -> verify chain completeness -> verify each chunk hash on read -> verify reconstructed-file hash, with a defined failure action. Define quarantine concretely for each storage backend (mark restore-point status in SaaS DB + skip in restore selection when storage is append/write-only). Specify the structured restore audit event schema NOW (timestamp, actor, device, restore-point id, integrity result, bytes restored) and emit it from the Sprint 1 structured-logging foundation, even though full audit trail is Sprint 8 — the field set is expensive to add retroactively.

### RST-8 — No restore-verification / test-restore automation, so backup success is reported without ever proving restorability
`Medium severity · Medium likelihood`

- **Risk:** The roadmap (Sprint 9 Reporting) covers alerts and storage warnings but there is no automated test-restore or recovery-verification anywhere in the 10 sprints. The agent reports a backup as successful based on upload completion, not on a proven ability to reconstruct the data. This contradicts the 'restore correctness first' principle: success is measured on the write path, never the read path.
- **Impact:** Backups are green for months, then the one real restore reveals systemic corruption (bad chunking, key drift, manifest bug) affecting an entire customer base at once — the worst possible time to discover it. SMBs/MSPs who trust the dashboard get a false sense of safety.
- **Recommendation:** Add an automated recovery-verification capability to the restore engine roadmap: periodic sampled test-restores (decrypt + decompress + hash-verify a random subset of chunks/files against stored hashes) reported as a distinct 'last verified restorable' status in the SaaS panel, separate from 'last backup succeeded'. Build the per-chunk and per-file hashes needed to support this into the manifest format from Sprint 3 (decided now), so verification can be added without re-reading or re-uploading data.

### RST-9 — Cross-version restore (old backup, new agent) has no compatibility contract — manifest/chunk format created in Sprint 3 with no forward/backward versioning
`Medium severity · Medium likelihood · **S1**`

- **Risk:** Agent auto-update is a core feature (PRD, updater in Sprint 1), so agents WILL upgrade while old backups persist in storage for the retention period. The manifest and chunk/encryption formats are created across Sprints 3-5 with no stated versioning or compatibility policy. Combined with the ambiguous hash algorithm (RST-3) and unversioned manifest (RST-2), an upgraded agent may be unable to restore backups it took months earlier.
- **Impact:** An auto-update silently breaks restorability of all pre-update backups, which is the exact set a customer reaches for during a disaster (they want last month's clean copy after a ransomware event). The auto-update feature thus becomes a latent data-availability risk.
- **Recommendation:** Establish a format-versioning and compatibility contract NOW: every manifest, chunk container, and encrypted-object header carries an explicit format-version, and the restore engine must support reading all prior versions within the supported retention window (define an explicit support window, e.g. N major versions / X months). Add a cross-version restore test matrix to CI before Sprint 6 ships. The version fields themselves must be present in the very first format Sprint 3 emits — a Sprint 1 design decision.

## 6. Agent-Update Risks

*The auto-update path is the single most dangerous subsystem in Beyz Backup because a bad update can brick the agent on thousands of unattended SMB/hotel/MSP machines at once, and unlike a failed backup it has no human in the loop to catch it. The docs treat update signing as a deferred concern: SECURITY.md and ROADMAP.md push "Signed Updates" to Sprint 8, while the Sprint 1 updater ships as "placeholder logic" for signature verification, SHA256, binary replacement, and rollback. This directly collides with CLAUDE.md's rules ("Do not create placeholder security implementations in production code", "Do not remove update signature validation"). The truly load-bearing decisions — which signature algorithm, where the private signing key lives, how it rotates/revokes, the manifest wire format and versioning, the self-update mechanics on a locked running Windows Service, and what rollback actually restores — are all unspecified and all extremely expensive to retrofit once agents are deployed in the field. The updater is also the most attractive target in the whole system: code-execution-as-SYSTEM on every endpoint. Several of these must be designed for in Sprint 1 even though the feature fully ships later, because the manifest format, key custody model, and version/downgrade scheme become wire-compatibility and trust-anchor commitments the moment the first agent is installed.*

### UPD-1 — Signing-key custody, algorithm, and trust anchor are undefined but get baked into every installed agent in Sprint 1
`Critical severity · High likelihood · **S1**`

- **Risk:** SECURITY.md requires updates to be signed and verified, but never states the signature algorithm, where the private key lives, or how the agent obtains the trust anchor (public key). ROADMAP.md defers 'Signed Updates' to Sprint 8, yet the Sprint 1 installer ships beyz-backup-updater.exe and the Inno Setup installer lays down the agent. The public verification key must be embedded in (or pinned by) the agent/updater at install time. Whatever trust anchor ships in the first installer becomes the root of trust for every field device. CLAUDE.md explicitly forbids 'placeholder security implementations in production code' and 'remove update signature validation', so a stub that skips verification cannot ship — but the docs give no concrete scheme to implement instead.
- **Impact:** If the algorithm or key location is chosen ad hoc later (e.g. signing key on a build server, or an RSA key with no rotation), an attacker who compromises the build/CI pipeline or that single key gains code-execution-as-SYSTEM on every enrolled endpoint via a forged update — full ransomware/backdoor distribution to all customers at once. Retrofitting a different trust anchor into already-installed agents requires a flag-day re-enrollment.
- **Recommendation:** Decide and document NOW (Sprint 1): use Ed25519 for the update signature (small, fast, no padding pitfalls). Generate the private signing key in an offline HSM or cloud KMS (e.g. AWS KMS / YubiHSM) that the build pipeline can only invoke, never export. Embed the Ed25519 public key as a compile-time constant in the updater binary so the trust anchor cannot be swapped at rest. Treat the signing key as a separate, higher-trust key than any TLS/API key. Even though signed updates 'ship' in Sprint 8, the public key and verification code path must be present and enforcing in the Sprint 1 binary so the first installed agent already has the correct trust anchor.

### UPD-2 — No key rotation or revocation design — a single embedded public key is unrotatable once agents are in the field
`Critical severity · Medium likelihood · **S1**`

- **Risk:** SECURITY.md lists only 'All updates must be signed' with no mention of key rotation or revocation. If a single Ed25519/RSA public key is embedded in the updater and the corresponding private key is later compromised, there is no way to revoke it or roll to a new key — because the only mechanism to update the agent (the updater itself) is exactly what trusts the compromised key. This is a chicken-and-egg trap that is unfixable post-deployment without manual reinstall on every device.
- **Impact:** A leaked signing key becomes permanent: the attacker can push malicious signed updates forever, and the vendor cannot revoke it remotely because the revocation would itself have to be delivered through the trusted-update channel the attacker now also controls. For a backup product (SYSTEM-level access to all customer data + storage credentials), this is a worst-case, business-ending compromise.
- **Recommendation:** Design a multi-key trust model in Sprint 1: embed a set of N public keys (or an offline root key that signs short-lived signing-subkeys, like a mini-PKI / Sigstore-style approach). Manifests carry a key-id; the agent accepts a signature from any non-revoked embedded key. Ship a signed 'key-revocation list' field in the manifest format from day one (even if empty), and define an out-of-band recovery path (e.g. installer re-download with a new pinned key) for total-compromise scenarios. These are format and key-material decisions that must exist before the first installer is published.

### UPD-4 — Rollback is a placeholder with no defined trigger, health check, or 'what gets restored' contract
`Critical severity · High likelihood · **S1**`

- **Risk:** PRD requirement 12 and ARCHITECTURE.md list 'Rollback' as an updater responsibility, but nothing defines: what triggers a rollback, what a successful update looks like (post-update health check), how long to wait before declaring failure, or what state rollback restores (just the .exe? config too? the enrollment cert?). CLAUDE.md states 'Restore reliability has higher priority than backup speed' and 'Restore correctness first' — the same principle must apply to update rollback, but the design is absent. A rollback that doesn't fire on the right signal, or that restores a binary while leaving a migrated config, is worse than no rollback.
- **Impact:** An update that installs cleanly but then crash-loops on startup (bad config migration, missing dependency, AV quarantine of the new exe) will never trigger rollback if 'success' is defined as 'service start returned' rather than 'agent reached a healthy steady state and checked in'. Result: fleet-wide silent agent death after a routine update, with no automatic recovery.
- **Recommendation:** Define a concrete post-update health gate in Sprint 1: after restart, the new agent must (a) start, (b) successfully send one heartbeat / self-report 'update OK' to the SaaS within a bounded timeout (e.g. 90s), before the updater commits. If the gate fails, the updater automatically restores the previous binary AND the previous config snapshot from the .bak set and restarts. Specify exactly what the update bundle versions together (binary + config schema) so rollback restores a consistent pair. Wire the heartbeat ack into the update success signal — the heartbeat sender is already Sprint 1 scope, so this is cheap to design in now.

### UPD-3 — Self-updating a running Windows Service: locked .exe and the updater-updating-itself problem are unaddressed
`High severity · High likelihood · **S1**`

- **Risk:** PRD requirement 12 lists 'stopping agent service', 'replacing binary', 'restarting service' as updater steps, but the docs ignore Windows file-locking semantics. A running .exe is locked by the OS; you cannot overwrite beyz-backup-agent.exe while the service runs, and beyz-backup-updater.exe cannot overwrite itself while it is the process performing the update. The naive 'stop service -> copy new exe -> start service' sequence is fragile: if the updater crashes or the machine loses power between stop and successful start, the device is left with a stopped service and possibly a half-written binary — a bricked agent with no human present.
- **Impact:** On unattended SMB/hotel/MSP endpoints, a power loss or AV interference mid-replace leaves the backup agent permanently down until someone physically reinstalls it. Silent backup gaps on machines that were supposedly protected — discovered only when a restore is needed. For self-update of the updater itself, a botched swap means no future updates can ever be applied (no recovery channel).
- **Recommendation:** Design the replacement mechanic in Sprint 1: (1) write the new binary to a temp name, verify signature+hash, then do an atomic MoveFileEx rename into place rather than an in-place overwrite; (2) for updater self-update use the classic two-stage pattern — the running updater spawns a tiny bootstrap copy (or uses MoveFileEx with MOVEFILE_DELAY_UNTIL_REBOOT as a fallback) so no process overwrites its own running image; (3) require the service to confirm a successful post-start health check before the old binary (.exe.bak) is deleted. Define this state machine and the on-disk layout (current/new/backup) now, since the installer and directory structure are being created in Sprint 1. **Sprint-1 scope (Technical Design §0.6):** ship the *agent-binary* atomic swap + health-gated rollback for real; the *updater self-update* two-stage bootstrap and the persistent watchdog are **deferred to Sprint 8** (the updater is an on-demand binary, not a second service).

### UPD-5 — No staged/canary rollout, version pinning, or kill-switch — a bad update goes to 100% of the fleet instantly
`High severity · Medium likelihood · **S1**`

- **Risk:** The PRD describes 'Agent auto-update system' (item 8) as a fleet capability but the docs contain no concept of staged rollout, canary cohort, per-tenant version pinning, or a kill-switch to halt a rollout in progress. The manifest check in the updater (PRD req 12) is implicitly 'is there a newer version? then take it.' Multi-tenant SaaS for MSPs means one bad manifest fans out to every device across every customer simultaneously.
- **Impact:** A regression that passes QA but fails on some real-world configuration (a specific Windows build, AV, locale) takes down a large fraction of the fleet before anyone notices, because there is no ramp and no brake. For an MSP whose entire client base runs Beyz agents, a single bad push is a mass-outage and a contractual/reputational catastrophe. Without a kill-switch, even after detecting the problem the vendor cannot stop in-flight updates.
- **Recommendation:** Bake rollout-control fields into the manifest/version-check protocol in Sprint 1 even though orchestration ships later: the SaaS update-check response (the task-polling/manifest channel) should carry target_version + a rollout cohort/percentage + an explicit 'update_allowed' boolean (kill-switch) per device. The agent only updates when the server tells it to, rather than the agent autonomously pulling 'latest'. Support per-tenant version pinning server-side. Designing the protocol as server-directed now avoids a wire-format break later.

### UPD-6 — No downgrade protection — signed-but-old binaries enable rollback attacks
`High severity · Medium likelihood · **S1**`

- **Risk:** SECURITY.md requires signature + SHA256 + version-manifest verification but never says the agent must reject older versions. A correctly signed older binary (e.g. a previous release with a known, now-patched vulnerability) will pass all three checks. Without monotonic version enforcement, an attacker who can influence the manifest channel (or replay an old signed manifest) can force-downgrade agents to a vulnerable build and then exploit it.
- **Impact:** Security patches become non-sticky: an attacker rewinds the fleet to a vulnerable agent version using only legitimately signed artifacts, defeating the entire point of having shipped the patch. Particularly dangerous for a SYSTEM-level backup agent that holds storage credentials.
- **Recommendation:** Make the signed manifest include a monotonic version number and a 'minimum acceptable version' floor, and have the agent persist its current version and refuse any update whose version is <= current (except an explicit, separately-flagged emergency-downgrade signed by the offline root key). Anti-rollback comparison must use the signed manifest's version, never the filename or a server header. This is a manifest-format and agent-state decision, so it must be fixed in Sprint 1.

### UPD-7 — Manifest format and transport are undefined, risking a non-versioned, non-extensible wire contract
`High severity · High likelihood · **S1**`

- **Risk:** ARCHITECTURE.md ('Manifest Validation', 'Version Check') and SECURITY.md ('Version Manifest') reference a manifest but never define its format, fields, how the signature covers it, or its transport (HTTPS endpoint? S3 object?). If Sprint 1 ships an ad-hoc manifest (e.g. just version+url+sha), it omits the very fields the later sprints need (key-id for rotation per UPD-2, version floor per UPD-6, rollout cohort/kill-switch per UPD-5, per-platform binaries for the Linux systemd agent). Once agents in the field parse a given manifest shape, changing it is a breaking change.
- **Impact:** A too-thin manifest forces an incompatible v2 format later; old field agents can't parse it, so the update channel needed to fix them is exactly what breaks. Also, if the signature only covers the binary hash and not the full manifest (version, URL, flags), an attacker can tamper with unsigned manifest metadata even while the binary signature stays valid.
- **Recommendation:** Specify a versioned, canonical manifest in Sprint 1 (e.g. a JSON document with an explicit schema_version, reserved fields for key_id, min_version, rollout, kill_switch, and a per-platform artifacts array). The Ed25519 signature must cover the entire canonicalized manifest, not just the binary hash. Transport over HTTPS with cert pinning to the SaaS, and treat the manifest as immutable/cache-busted. Reserve the extension fields now even if unused until Sprint 5/8.

### UPD-9 — No watchdog to recover a dead agent — single failure point with no second line of defense
`High severity · Medium likelihood · **S1**`

- **Risk:** The design has the updater stop/replace/restart the agent, but nothing watches the watcher. If the agent service is down (failed update, crash-loop, AV quarantine) and the updater is not currently running (it ran once and exited), there is no independent component left to detect 'agent has been down for N minutes' and trigger recovery or rollback. Windows Service recovery actions (SC failure restart) restart the same possibly-broken binary, not the previous good one. On unattended endpoints this means a down agent stays down indefinitely.
- **Impact:** Backups silently stop on affected machines with no automatic recovery and no local human; the SaaS may show 'no heartbeat' but cannot fix it remotely if the only management channel (the agent) is the thing that's dead. This converts a transient update glitch into a permanent protection gap discovered at restore time.
- **Recommendation:** Introduce an independent watchdog in the Sprint 1 architecture (it can be the updater installed as a separate lightweight always-on service or a scheduled task) whose only job is: detect prolonged agent absence/health-check failure and trigger rollback to .exe.bak or re-fetch a known-good version. Configure Windows Service recovery as a backstop, but the authoritative recovery must compare against the last-known-good binary, not just restart the current one. Deciding that the updater is a persistent service vs a one-shot process is an installer/service-registration decision that must be made in Sprint 1.

### UPD-8 — SHA256-vs-signature redundancy gives a false sense of integrity if the hash isn't inside the signed manifest
`Medium severity · Medium likelihood · **S1**`

- **Risk:** SECURITY.md lists both 'Signature' and 'SHA256' as independent checks. SHA256 alone provides no authenticity — only an integrity check against a value the attacker also controls if that value is fetched separately or unsigned. The redundancy is only meaningful if the SHA256 of the binary is a field inside the signed manifest (so the signature authenticates the hash, and the hash lets you verify a separately-downloaded large binary without re-validating a signature over the whole blob). The docs don't state this relationship, risking an implementation where SHA256 is checked against an unsigned source — security theater.
- **Impact:** Developers may implement 'download binary, compute SHA256, compare to a hash from the download URL/header' and believe they have tamper protection, when in fact an attacker who controls the binary also controls the advertised hash. The signature check becomes the only real defense and any weakness in how it's applied is uncaught.
- **Recommendation:** Document the exact verification order in Sprint 1: (1) fetch manifest, (2) verify Ed25519 signature over the whole manifest, (3) read the expected SHA256 (or BLAKE3 per CLAUDE.md) of the binary FROM the now-trusted manifest, (4) download the binary, (5) verify its hash equals the manifest value, (6) only then proceed to atomic replace. The hash must never come from any source other than the signed manifest. Consider using BLAKE3 to stay consistent with CLAUDE.md's preferred hash.

## 7. Licensing Risks

*Licensing is a core monetization pillar of Beyz Backup (device-based + storage-quota per CLAUDE.md 14-15 and PRD.md 13-14), yet it is the single most under-specified part of the design: ARCHITECTURE.md (line 19) names "Licensing" as a SaaS responsibility with zero detail, and ROADMAP.md has NO licensing sprint across all 10 sprints. Meanwhile the very primitives any licensing scheme must bind to — the agent fingerprint, the single-use enrollment token, the agent certificate, the heartbeat, and the audit/tamper-log — are all being built NOW in Sprint 1 (PRD.md 58-68). This creates a dangerous asymmetry: the enforcement substrate is frozen in Sprint 1 with no licensing requirements driving its shape, so foundational decisions (fingerprint stability, enforcement locus, signed-license format, quota accounting source-of-truth, tenant scoping) will be made implicitly and be expensive to retrofit. Compounding this, the zero-knowledge encryption + ZSTD compression + chunk dedup architecture (SECURITY.md 8-10, ARCHITECTURE.md data flow) structurally prevents the server from measuring plaintext data size, directly undermining storage-quota accuracy. Most identified risks are Sprint-1-relevant because they constrain the fingerprint scheme, enrollment/cert protocol, config/secret layout, and audit-log design that Sprint 1 hard-codes.*

### LIC-1 — No licensing design exists, yet Sprint 1 freezes every primitive licensing must bind to
`High severity · High likelihood · **S1**`

- **Risk:** Device-based and storage-quota licensing are named as core product features (CLAUDE.md 14-15, PRD.md 13-14) and as a SaaS responsibility (ARCHITECTURE.md 19), but there is no licensing data model, no enforcement spec, no signed-license format, and crucially NO licensing sprint anywhere in ROADMAP.md's 10 sprints. At the same time, Sprint 1 (PRD.md 58-68) hard-codes the exact primitives any license scheme must attach to: agent fingerprint, single-use enrollment token, agent certificate, heartbeat, and audit log. Building these with no licensing requirements means their shape is decided implicitly.
- **Impact:** When licensing is finally designed (Sprint 9 'Storage Warnings' is the first hint, but enforcement is never scoped), the team discovers the Sprint 1 fingerprint isn't stable enough to bind a device license to, the enrollment token carries no license/tenant/quota claims, the heartbeat has no field to report license state or usage, and the audit log doesn't record license events. Retrofitting forces an agent-protocol version bump and re-enrollment of every deployed agent in the field — a costly, customer-visible migration that breaks exactly the SMB/MSP installs the product targets.
- **Recommendation:** Before writing Sprint 1 code, produce a one-page licensing contract pinning down: (1) the license claim set (tenant_id, device_id, device_count_limit, storage_quota_bytes, plan, expiry, grace_period, signature); (2) that the enrollment response returns a signed license blob the agent persists; (3) reserved fields in the heartbeat payload for license_state and reported_usage_bytes; and (4) license-event types in the audit log (issued/renewed/revoked/over-quota). Treat these as forward-compatible reserved fields in the Sprint 1 protocol even though enforcement ships later.

### LIC-2 — Agent fingerprint stability is unspecified — device-license binding will break on VM clone, re-image, and hardware change
`High severity · High likelihood · **S1**`

- **Risk:** PRD.md (line 63) lists 'Agent fingerprint' as a Sprint 1 deliverable but gives zero definition of what it hashes (MAC? motherboard serial? disk UUID? Windows MachineGUID? a generated install-time GUID?). Device-based licensing (CLAUDE.md 14) must count and bind to distinct devices, which requires a fingerprint that is both stable (survives reboots/updates) AND unique/non-clonable. These goals conflict and the doc resolves neither.
- **Impact:** If the fingerprint is hardware-derived: VM clones and golden-image/re-image deployments (extremely common in hotels and MSP-managed fleets — the named target market) produce duplicate or shifting fingerprints, causing legitimate devices to either collide (two machines counted as one, under-counting license) or be rejected after re-image (over-counting, false license-exhaustion lockouts that block backups). If it's a random install-time GUID stored in config.yaml: it's trivially clonable by copying the file, so the device-count cap is bypassable by image duplication, directly defeating per-device billing.
- **Recommendation:** Define the fingerprint NOW as a server-issued opaque device_id bound at enrollment to the single-use token, NOT a client-computed hardware hash. Persist it inside the agent certificate (already generated post-enrollment per SECURITY.md 18-23) so a cloned config.yaml without a matching server-side cert/device record is detectable. Additionally collect 2-3 hardware signals (MachineGUID + disk serial) reported to the server purely for clone-detection heuristics and re-enrollment matching, not as the primary key. Document the explicit re-image/clone re-enrollment flow.

### LIC-3 — Storage-quota licensing is structurally unmeasurable under zero-knowledge encryption + compression + dedup
`High severity · High likelihood · **S1**`

- **Risk:** Storage-quota licensing (CLAUDE.md 15, PRD.md 14) requires the server to meter tenant storage consumption. But the architecture mandates client-side encryption where 'Server must never access plaintext files' (SECURITY.md 8-10), ZSTD compression and chunk-based dedup with reference counting (ARCHITECTURE.md data flow; PRD.md 46-52). The server cannot see plaintext size, and post-compression/dedup stored bytes depend on cross-device and cross-restore-point dedup that only the storage layer knows. There is no defined 'unit' the quota is denominated in (plaintext GB? stored ciphertext GB? pre-dedup logical GB?).
- **Impact:** Quota enforcement and billing become ambiguous and disputable. If quota = stored ciphertext bytes, dedup means two customers backing up identical data are billed unequally and a customer can be punished/rewarded by dedup they don't control. If quota = client-reported plaintext size, the agent (untrusted, see LIC-5) is the meter and can under-report to evade the cap. For customer-owned storage targets (SMB/SFTP/S3/MinIO), the bytes never land in Beyz infrastructure, so the server has no independent measurement at all — quota is whatever the agent claims.
- **Recommendation:** Decide the quota unit explicitly and write it into the licensing contract: recommend metering on stored-ciphertext-bytes-after-dedup reported by the storage layer for Beyz Cloud, and a signed agent-side usage report (manifest byte totals) for customer-owned storage, cross-checked against storage-target listing where possible. For Sprint 1, reserve a 'reported_usage_bytes' field in the heartbeat and design the future manifest to carry per-restore-point logical and stored byte counts so the server can reconcile rather than trust a single number.

### LIC-4 — Enforcement locus and offline/grace-period behavior are undefined — air-gapped backups will silently fail or be silently free
`High severity · High likelihood · **S1**`

- **Risk:** The product targets multi-location companies and SMBs with intermittent or air-gapped connectivity, but nothing in any doc states WHERE license enforcement happens. The only server contact points in Sprint 1 are enrollment, heartbeat, and task polling (PRD.md 58-66). Agent-side enforcement is bypassable; server-side enforcement requires connectivity the target environments may lack. There is no grace period, no offline-validity window, and no defined behavior when the heartbeat can't reach the SaaS.
- **Impact:** Two failure modes, both bad. (a) Hard server-side gating on heartbeat: a transient outage or air-gapped site stops backups entirely — catastrophic for a backup product whose whole value is data protection, and a likely SLA/contract breach. (b) Agent runs freely offline with no signed time-bounded license: a customer can permanently air-gap the machine and use it license-free forever, and an attacker can block heartbeats to evade revocation (see LIC-7). The CLAUDE.md rule 'Restore reliability has higher priority than backup speed' implies availability must not be sacrificed, yet a naive online check does exactly that.
- **Recommendation:** Adopt a signed, time-bounded offline license: enrollment/heartbeat returns a license blob valid for N days (e.g. 14-30) carrying device/quota claims and an expiry. The agent enforces locally against this signed blob (cannot be forged without the server's private key), keeps backing up through outages until expiry, and refreshes on each successful heartbeat. Define and document post-expiry behavior (warn-then-degrade, never instant hard-stop of restores). Pin the license-signing keypair custody decision now — it is the root of trust for all enforcement.

### LIC-5 — Agent-side license/quota state has no anti-tamper design despite 'tamper detection' being a Sprint 1 line item
`High severity · Medium likelihood · **S1**`

- **Risk:** config.yaml lives in plaintext at C:\ProgramData\BeyzBackup (PRD.md 28-33), a path writable by local admins. PRD.md (line 68) lists 'Tamper detection' but ROADMAP.md never schedules it and no doc says what it protects. Any locally stored license state, device_id, quota counters, or cached 'days remaining' is editable by the customer, who has every incentive to extend expiry, raise the device cap, or zero the usage counter. The Go agent binary in C:\Program Files\BeyzBackup can also be patched.
- **Impact:** Device-count caps and storage quotas become honor-system on the very machines whose owners pay per device/GB. An admin edits config.yaml to flip the plan field or extend expiry and gets unlimited backup for free; with no server-side reconciliation (LIC-3, LIC-4) this is undetectable. Revenue leakage scales with sophistication of the customer base — MSPs and IT-savvy SMBs are exactly the ones who will find this.
- **Recommendation:** Never store enforceable license state as plaintext config the agent trusts. Store only a server-signed license blob whose signature the agent verifies against an embedded public key on every load; reject/re-enroll on signature failure. Keep the authoritative device count and quota usage SERVER-side, treating agent reports as advisory and reconciling against heartbeat presence and storage-layer measurements. Define the Sprint 1 audit log to record license-blob signature failures and config-tamper events as security events (SECURITY.md 57-68 already mandates security-event logging).

### LIC-6 — MSP multi-tenant sub-licensing has no model — enrollment token carries no tenant/quota scope
`Medium severity · Medium likelihood · **S1**`

- **Risk:** MSPs are a named primary market (PRD.md 5) and the platform is 'Multi-tenant first' (CLAUDE.md 72), implying an MSP must allocate device/quota sub-licenses across downstream customer tenants. Yet the Sprint 1 enrollment token (SECURITY.md 16-20, single-use) and registration placeholder (PRD.md 64) have no defined fields binding an agent to a specific tenant, parent MSP, or sub-license pool. The licensing hierarchy (MSP master license -> per-customer sub-allocation -> per-device consumption) is entirely absent.
- **Impact:** Without tenant scoping baked into enrollment, an agent enrolled under one MSP customer could be mis-attributed, and the MSP cannot enforce or report per-customer device/quota consumption — breaking both billing and the multi-tenant isolation the architecture promises. Retrofitting tenant/sub-license claims into the enrollment+certificate flow after agents are deployed forces re-enrollment (same migration pain as LIC-1/LIC-2).
- **Recommendation:** Define the enrollment token in Sprint 1 as a signed/opaque token that encodes (or maps server-side to) tenant_id and parent_account_id, and have the issued agent certificate embed the resolved tenant binding. Model the MSP license as a parent pool with allocatable device/quota sub-buckets, and ensure heartbeat usage reports are attributable up the tenant tree. Even if MSP UI ships later (Sprint 2 SaaS Core), the enrollment/cert binding must be tenant-scoped from the first agent.

### LIC-7 — License revocation, transfer, and theft have no enforcement path given heartbeat-only connectivity
`Medium severity · Medium likelihood · **S1**`

- **Risk:** No doc defines how a license is revoked (non-payment, security incident), transferred (hardware replacement, device retirement freeing a slot), or invalidated on theft (stolen agent cert/config). The only server->agent channel in Sprint 1 is heartbeat/task-polling (PRD.md 65-66), and the agent certificate is generated once at enrollment (SECURITY.md 18-23) with no stated rotation, expiry, or revocation list.
- **Impact:** Revocation is unenforceable against an agent that simply stops heartbeating (also the trivial offline-evasion path from LIC-4). A retired/re-imaged device permanently consumes a device slot if there's no transfer/release flow, causing false license exhaustion and support load. A stolen agent certificate grants indefinite backup/restore access to a tenant's encrypted store with no way to invalidate it — a security as well as licensing problem (restore access to customer data).
- **Recommendation:** Give the agent certificate a finite lifetime and require renewal via heartbeat so revocation = stop renewing (works with the offline-validity window from LIC-4, bounding theft/non-payment exposure to one grace period). Define explicit device-release/transfer API semantics that free a device slot server-side. Maintain a server-side revoked-device list checked at heartbeat and on every storage-layer access. Decide cert lifetime and renewal cadence in Sprint 1 since the cert is issued in Sprint 1.

### LIC-8 — Over-quota behavior policy is undefined — risk of silently dropping backups (data-loss) or unbounded free overage
`Medium severity · Medium likelihood`

- **Risk:** Neither PRD, ARCHITECTURE, nor ROADMAP states what happens when a tenant exceeds its storage quota or device count. The only related item is Sprint 9 'Storage Warnings' (ROADMAP.md 128-133), which is a notification, not an enforcement policy. The choice — hard-block new backups vs. allow-and-bill-overage vs. throttle — is a product/billing decision with major safety implications for a backup tool.
- **Impact:** If over-quota hard-blocks backups, a customer who outgrows their plan silently stops being protected and discovers it only after a data-loss event — the worst possible outcome for a backup product and a reputational/legal landmine. If it silently allows overage with no metering reconciliation (see LIC-3), Beyz absorbs uncontrolled storage cost, especially on Beyz Cloud. Building the agent and manifest without a quota-enforcement hook means there's nowhere clean to insert either policy later.
- **Recommendation:** Decide the over-quota policy explicitly (recommend: never block restores; for backups, warn at 80/100%, then soft-grace overage with billing rather than instant data-protection loss, with a hard ceiling). Ensure Sprint 1's agent/manifest design reserves a server-driven 'backup_allowed' / 'quota_status' flag in the heartbeat/task-polling response so enforcement can be toggled centrally without an agent update. Log quota-threshold and over-quota events to the audit log.

## 8. Storage Risks

*The storage design is the highest-risk and least-specified dimension of Beyz Backup. Five named targets (SMB, SFTP, S3, MinIO, Beyz Cloud) are listed in ARCHITECTURE.md with zero abstraction-interface definition; the storage layer is deferred to Sprint 7 while the chunking/encryption pipeline that depends on it lands in Sprints 3-6, creating an inverted dependency order. The most consequential gaps are foundational and must be decided in Sprint 1 even though storage code ships later: where and how customer storage credentials are stored, what a chunk's storage key/naming scheme is (it must encode the encryption/hash scheme and a tenant boundary), and whether cross-tenant chunk deduplication is allowed (it breaks zero-knowledge and reference counting). The docs also make implicit durability promises ("backup", restore correctness "first") that customer-owned storage physically cannot keep — customers can delete or expire their own data, and SMB/SFTP cannot enforce the immutability that the ransomware-recovery value proposition implies. Reference counting, garbage collection of orphaned chunks, S3 multipart/resumable-upload integrity, and restore egress cost are all unaddressed. Several of these are cheap to design now and very expensive to retrofit once the on-disk chunk layout and credential format are shipped to customers.*

### STO-1 — Customer storage credentials have no defined storage location, encryption, or custody model
`Critical severity · High likelihood · **S1**`

- **Risk:** Targets SMB, SFTP, S3, MinIO all require long-lived secrets (SMB user/password, SFTP key/password, S3 access-key/secret-key, MinIO creds). ARCHITECTURE.md lists these five targets but no document specifies WHERE these credentials live (SaaS DB? agent config.yaml at C:\ProgramData\BeyzBackup? both?), how they are encrypted at rest, or who can decrypt them. CLAUDE.md only says 'never hardcode secrets' / 'use environment variables' — inadequate for per-tenant, per-target credentials that the agent needs at runtime. SECURITY.md's zero-knowledge claim ('Server must never access plaintext files') is silent on storage credentials, which are a separate secret class. config.yaml lives in C:\ProgramData which is world-readable by default on Windows.
- **Impact:** If credentials sit in plaintext config.yaml, any local user or malware reads them and gains full write/delete access to the customer's entire backup repository — enabling deletion of all backups (the exact ransomware scenario the product is sold to prevent). If they sit unencrypted in the SaaS DB, a SaaS breach hands an attacker every tenant's storage. Retrofitting credential encryption after config.yaml format ships to customers requires a migration on every deployed agent.
- **Recommendation:** In Sprint 1 define a CredentialStore abstraction and the config schema slot for it even though no target is wired up yet. Decide custody now: store customer storage credentials in the SaaS DB encrypted with a per-tenant KMS/envelope key, deliver them to the agent over the authenticated channel just-in-time (not persisted in config.yaml), and if local persistence is unavoidable use Windows DPAPI (machine+service-account scope), not plaintext YAML. Lock down C:\ProgramData\BeyzBackup ACLs to the service account in the Sprint 1 installer. Document that storage credentials are a distinct secret class from the file-encryption key.

### STO-4 — Reference counting + multi-tenant shared chunks risks zero-knowledge breach and cross-tenant data deletion
`Critical severity · High likelihood · **S1**`

- **Risk:** CLAUDE/PRD specify chunk-based backup with 'Reference counting later' (Sprint 5). Chunk dedup typically keys chunks by content hash (BLAKE3/SHA256 per CLAUDE). If dedup is global/cross-tenant on the SaaS-managed storage, two tenants with the same plaintext block share one stored chunk — but client-side AES-256-GCM encryption (zero-knowledge) means identical plaintext only dedups if tenants share keys or use convergent encryption, which leaks plaintext-equality across tenants and breaks the zero-knowledge promise in SECURITY.md. Conversely, if the refcount table is shared, a delete by tenant A could decrement and GC a chunk tenant B still references (a confused-deputy GC bug = silent data loss).
- **Impact:** Either a confidentiality breach (cross-tenant plaintext-equality leakage / shared encryption keys violating 'Server must never access plaintext') or catastrophic silent data loss (tenant B's restore points lose chunks GC'd on tenant A's behalf). Both are unrecoverable trust failures. The chunk storage key scheme is baked in at the first byte written and cannot be changed without rewriting every customer's repository.
- **Recommendation:** Decide the dedup boundary in Sprint 1 and encode it in the chunk naming scheme: scope dedup and reference counting PER TENANT (or per encryption-key domain), never globally, so client-side encryption stays zero-knowledge and refcounts can never cross a tenant boundary. Define the chunk storage key format now as e.g. tenant_id/encryption-scheme-version/hash, and make tenant_id a hard prefix that also serves as the storage-side isolation boundary. Explicitly forbid convergent/cross-tenant dedup in the design doc.

### STO-2 — No storage abstraction interface defined before pipeline consumers are built
`High severity · High likelihood · **S1**`

- **Risk:** ARCHITECTURE.md lists five storage targets as a flat bullet list with no common interface. The data flow (chunk->hash->compress->encrypt->upload) places 'Upload' as a single step, but SMB (file copy, no native multipart/resume), SFTP (streaming, partial-file append semantics), and S3/MinIO (object PUT, multipart, ETag/object-lock) have fundamentally different capabilities. ROADMAP puts Backup Engine (Sprint 3) and Upload before Storage Expansion (Sprint 7), so the upload caller will be coded against whatever ad-hoc S3 logic exists first, then bent to fit SMB/SFTP later.
- **Impact:** Without a capability-aware interface (Put/Get/Delete/List + capability flags for multipart, resumable, immutability, atomic-rename), the backup engine will hardcode S3 assumptions in Sprint 3 and require invasive rework in Sprint 7 to support SMB/SFTP — exactly the rework CLAUDE.md's Clean Architecture / SOLID principles are meant to prevent. Restore correctness ('first' priority) is jeopardized because each backend's failure/partial-write modes differ.
- **Recommendation:** In Sprint 1 write the Go StorageBackend interface (Put, Get, Delete, List, Stat, plus a Capabilities() returning flags: SupportsMultipart, SupportsResume, SupportsImmutability, SupportsAtomicRename, SupportsServerSideHash). Make the backup engine depend only on this interface. Provide a no-op/local-filesystem stub now so Sprint 3 codes against the abstraction, not against raw S3. This is a pure design artifact and costs little now but is very expensive to retrofit.

### STO-3 — Customer-owned storage cannot guarantee the durability/availability the product implies, and SMB/SFTP cannot enforce immutability
`High severity · Medium likelihood · **S1**`

- **Risk:** PRD/CLAUDE position the product as a Veeam/Acronis alternative and rank 'Restore correctness first'. But the hybrid model lets customers point backups at their own SMB/SFTP/S3/MinIO, where the customer (or their ransomware/admin/disgruntled employee) can delete, overwrite, or let storage fill up — with no immutability. SECURITY.md lists 'Immutable Storage' only as a FUTURE feature, and SMB/SFTP have no object-lock/WORM primitive at all, so they can never honor it. The marketing promise (ransomware protection) is structurally unachievable on the most common SMB targets.
- **Impact:** A customer suffers ransomware, the malware encrypts/deletes the SMB share holding backups, restore fails, and the customer blames Beyz despite Beyz never controlling that storage. This is both a support/liability nightmare and a trust-destroying event for the core sales pitch. Mixed expectations across targets (S3 object-lock works, SMB never will) create inconsistent guarantees the UI never communicates.
- **Recommendation:** Decide and document the durability/immutability tiering now, per target: S3/MinIO -> support object-lock/WORM when configured; SMB/SFTP -> explicitly classed as 'no immutability, customer-managed durability'. Surface a per-target capability badge in the SaaS UI and require explicit acknowledgement when a customer selects a non-immutable target for ransomware-class protection. Recommend (and later default) a Beyz Cloud immutable copy for true 3-2-1. This is a policy/abstraction decision (ties to STO-2 capability flags) best made in Sprint 1.

### STO-5 — Chunk object naming/key scheme is undefined and must encode encryption-scheme and hash-algorithm versions
`High severity · Medium likelihood · **S1**`

- **Risk:** CLAUDE allows 'BLAKE3 preferred, SHA256 acceptable' and AES-256-GCM, with manifests and restore points. Nothing defines how a chunk maps to a storage object name/path. If the chunk key is just the raw hash, the system cannot later distinguish a BLAKE3-named chunk from a SHA256-named one, cannot rotate the encryption scheme, and cannot validate which algorithm produced a given object on restore. ARCHITECTURE places hashing before encryption, so the stored object is ciphertext but named by plaintext-or-compressed hash — the relationship must be pinned down.
- **Impact:** Allowing two hash algorithms without versioning the key scheme guarantees a future migration headache: ambiguous object names, inability to verify integrity on restore (which hash do I recompute?), and inability to ever deprecate SHA256. Because object names are immutable once written to customer storage, this is a permanent on-disk format decision.
- **Recommendation:** In Sprint 1 freeze the chunk storage key format and put it in ARCHITECTURE.md: include a scheme/version tag, the hash algorithm id, and the tenant prefix, e.g. {tenant_id}/v1/blake3/{hash}. Pick ONE default hash algorithm for v1 (prefer BLAKE3) rather than leaving 'SHA256 acceptable' as a runtime choice; record the algorithm in the manifest per chunk so restore knows exactly what to verify. Make the manifest schema versioned from day one.

### STO-6 — Resumable/multipart upload integrity is asserted but has no defined verification or commit protocol
`High severity · High likelihood · **S1**`

- **Risk:** PRD lists 'Resumable upload' and 'Chunk upload' as transfer requirements, and ARCHITECTURE shows a single 'Upload' step. But there is no defined integrity/commit protocol: how a partially-uploaded multipart object is detected and resumed, how per-part and whole-object hashes are verified (S3 ETag is NOT a reliable whole-object hash for multipart), how a manifest is committed only after all chunks are durably persisted, and how SMB/SFTP — which lack multipart — emulate atomic commit (temp file + rename, where SFTP rename atomicity varies by server).
- **Impact:** Without an explicit commit/verify protocol, a crashed or resumed upload can leave a manifest pointing at chunks that are absent or half-written, producing restore points that fail at restore time — violating 'restore correctness first' and 'integrity verification before restore' (SECURITY.md). This is the classic backup-that-can't-be-restored failure that destroys backup products.
- **Recommendation:** Define the upload commit protocol now (it constrains the storage interface from STO-2): upload all chunks first, verify each with a client-computed BLAKE3/SHA256 (do not trust S3 ETag for multipart), then write the manifest LAST as the atomic commit point; on resume, list existing chunks and skip verified ones. For SMB/SFTP require write-to-temp-then-rename and detect non-atomic-rename servers via the capability flag. Add a 'manifest references a missing/corrupt chunk' integrity check to the restore path's quarantine logic.

### STO-7 — Orphaned-chunk garbage collection on customer-controlled storage has no safe design
`Medium severity · Medium likelihood`

- **Risk:** Sprint 5 adds 'Reference Counting' and 'Retention Policies', implying GC of unreferenced chunks. But the authoritative refcount/manifest state lives in the SaaS DB while the actual chunks live on storage the CUSTOMER controls and can mutate out-of-band (delete files, restore from a snapshot, change credentials). There is no design for reconciling DB-side refcounts with the real object listing, no mark-and-sweep safety window, and no handling of the case where the customer deletes chunks the manifest still needs.
- **Impact:** Aggressive GC based on a stale or out-of-sync refcount table deletes chunks still referenced by valid restore points (data loss). Conservative-but-unreconciled GC leaks orphaned chunks forever, silently inflating storage-quota billing (a licensing dimension) and customer cost. Out-of-band customer deletions leave dangling manifest references discovered only at restore time.
- **Recommendation:** Design GC as mark-and-sweep with a grace/quarantine window and a reconciliation pass that LISTs actual storage objects and compares against the manifest set before any delete — never delete on refcount alone. Treat the storage object listing as one source of truth and the SaaS refcount DB as the other, and alert on divergence rather than auto-deleting. Defer implementation to Sprint 5 but record the reconciliation requirement in ARCHITECTURE now so the manifest/refcount schema carries the needed object-listing metadata.

### STO-8 — Restore egress cost and S3 read/consistency behavior are unaccounted for in the licensing/restore model
`Medium severity · Medium likelihood`

- **Risk:** The product bills on device + storage-quota (PRD) but says nothing about restore EGRESS. For customer-owned S3 (esp. AWS) and Beyz Cloud, a full-device restore pulls every chunk back out — egress is often the most expensive line item and is not modeled in quota/licensing. SECURITY.md requires integrity verification before restore, meaning chunks may be downloaded (and possibly re-downloaded on a failed verify) amplifying egress. No mention of read consistency assumptions (modern S3 is strongly consistent, but MinIO/SMB/SFTP and older-compatible gateways vary) for the read-after-write the upload-then-manifest flow depends on.
- **Impact:** A large restore generates a surprise multi-hundred-dollar egress bill (on customer S3 or passed-through Beyz Cloud) with no forewarning, damaging trust at the worst possible moment (during a disaster recovery). Unverified consistency assumptions on non-AWS backends can cause a just-written manifest/chunk to read as missing, failing restores intermittently.
- **Recommendation:** Add restore egress estimation to the restore-initiation UX (show projected bytes/cost before a large restore) and decide now whether Beyz Cloud restore egress counts against quota or is billed separately. Document the read-after-write consistency requirement in the storage interface contract and gate non-strongly-consistent backends behind a retry-with-backoff read path. This is mostly Sprint 6+ but the consistency contract belongs in the Sprint 1 interface design.

## 9. Missing-Requirement Risks

*The five documents are dangerously underspecified for a production, zero-knowledge, multi-tenant backup SaaS. The PRD lists 24 Sprint 1 tasks at the implementation-detail level but the supporting docs (ARCHITECTURE, SECURITY, ROADMAP) are one-line bullet lists with no protocols, schemas, threat model, or non-functional targets. Critically, the very features that make Sprint 1 "agent foundation" — enrollment token, agent fingerprint, agent certificate, config/secret layout, heartbeat/poll protocol, signed-update trust root, audit log — are exactly the foundational decisions that are expensive to retrofit. Several are specified as contradictory or impossible (e.g., "zero-knowledge encryption" with no key custody/recovery design; "single-use enrollment token" with no issuance/expiry/revocation model; "audit log" listed both as a Sprint 1 security event source and a Sprint 8 deliverable). The whole product category (backup for SMBs/hotels/MSPs) implies RPO/RTO/SLA, retention/legal-hold, GDPR data residency, device decommission + crypto-shred, and bandwidth control — none of which appear anywhere. Many of these must be decided NOW because they shape the agent's config schema, enrollment protocol, fingerprint scheme, and on-disk credential format that Sprint 1 ships.*

### GAP-1 — Zero-knowledge encryption with no key management, escrow, or recovery design
`Critical severity · High likelihood · **S1**`

- **Risk:** CLAUDE.md and SECURITY.md mandate client-side AES-256-GCM with 'Server must never access plaintext' and CLAUDE.md forbids storing keys in source/server. Yet nowhere is it specified WHERE the data encryption key (DEK) comes from, how it is derived (passphrase? per-tenant master? per-device?), where it lives on the agent (the Sprint 1 config.yaml at C:\ProgramData\BeyzBackup), how it is wrapped, rotated, or — most importantly — recovered. ROADMAP defers 'Key Management' to Sprint 4 and 'Hardware-backed keys' to the future, but the agent's on-disk credential/secret layout is created in Sprint 1.
- **Impact:** In a true zero-knowledge model, a lost key = permanently unrecoverable backups. With no escrow/recovery design, a hotel or SMB that loses a device or passphrase loses ALL backups — a catastrophic, support-ticket-generating, reputation-ending failure for a backup product whose entire value is recoverability. Retrofitting key escrow later means re-encrypting every customer's data and migrating every agent's secret store.
- **Recommendation:** Before writing Sprint 1 config code, define the key hierarchy: per-tenant master key -> per-device DEK wrapped by it; document a recovery/escrow path (e.g., tenant master key escrowed in SaaS HSM/KMS with break-glass, or customer-held recovery key generated at enrollment and shown once). Specify the on-disk secret format and OS protection (Windows DPAPI/CNG, not plain config.yaml) so Sprint 1 ships the right credential store. Write down explicitly what 'zero-knowledge' means here (does the server hold a wrapped key it cannot unwrap, or nothing at all?).

### GAP-2 — Enrollment token / agent identity protocol is undefined (issuance, expiry, revocation, fingerprint scheme)
`Critical severity · High likelihood · **S1**`

- **Risk:** PRD req 7/18 and SECURITY.md require a 'single-use' enrollment token and an 'Agent Certificate generated after enrollment' plus an 'Agent fingerprint' (PRD security list). But nothing defines: token format/entropy, TTL/expiry, how it's bound to a tenant, what 'single-use' enforcement looks like (server-side nonce burn vs. client claim), what the fingerprint is computed from (hardware IDs? cert thumbprint?), how certificates are issued/renewed/revoked, or what the registration request/response wire format is. The SaaS registration is a 'placeholder' (req 8) yet the agent's enrollment handshake and identity are permanent Sprint 1 artifacts.
- **Impact:** A weak or replayable enrollment token lets an attacker enroll rogue agents into a victim tenant (data exfiltration vector in a multi-tenant SaaS). A fingerprint derived from mutable hardware (MAC/disk serial) breaks identity on hardware change or VM cloning, causing mass false 'new device' license consumption or lockouts. Getting the fingerprint scheme or cert lifecycle wrong forces a breaking re-enrollment of the entire installed base.
- **Recommendation:** Specify NOW: token = high-entropy (>=128-bit) opaque value, server-side single-use with explicit TTL (e.g., 24h) and tenant binding; define the registration JSON contract (request: token + fingerprint + agent version/OS; response: agent ID + client cert + poll/heartbeat endpoints). Define fingerprint as a stable, regenerable value (e.g., the public key of an agent-generated keypair) rather than scraped hardware IDs. Document cert renewal and revocation (CRL/short-lived certs) even if the SaaS side is stubbed in Sprint 1, so the agent stores and rotates the right artifact.

### GAP-3 — Software-update trust root and signing-key custody are unspecified
`Critical severity · Medium likelihood · **S1**`

- **Risk:** PRD req 11-12, SECURITY.md ('All updates must be signed', verify Signature + SHA256 + Version Manifest), and CLAUDE.md ('Do not remove update signature validation') mandate signed auto-updates. The updater is built in Sprint 1 (req 11). But there is no spec for: the signing algorithm/key type, where the public key/trust anchor is embedded (pinned in the binary? in config?), who holds the private signing key and how it's protected, manifest schema/version fields, anti-rollback/downgrade protection, or how the trust root itself is rotated if compromised.
- **Impact:** The auto-updater is the single highest-value attack surface in the product: a compromised or mis-designed update channel pushes attacker code to EVERY agent (a SolarWinds/supply-chain class event) running as SYSTEM via Windows Service. Without anti-rollback, an attacker serves an old vulnerable signed version. If the trust root isn't pinned in the Sprint 1 binary, there's nothing to anchor verification to; retrofitting key pinning requires a manual re-install of all agents.
- **Recommendation:** Decide and embed in Sprint 1: use Ed25519 or RSA-4096 signatures; pin the public key (or a small set, for rotation) inside the agent/updater binary, not in writable config. Define the manifest schema with explicit monotonic version + per-file SHA256, and enforce anti-downgrade (refuse versions <= current unless an explicit signed rollback flag). Document signing-key custody (offline/HSM, who can sign, audit of signing events) and a key-rotation procedure with overlapping trust. Treat the placeholder updater's verification code as real-from-day-one per CLAUDE.md's 'no placeholder security' rule.

### GAP-4 — No agent uninstall / device decommission / crypto-shred or offboarding flow
`High severity · High likelihood · **S1**`

- **Risk:** The Inno Setup installer (PRD req 13-18) covers install, folder creation, service registration, start, and token input — but nothing covers UNINSTALL, license release, secret wiping, or server-side de-enrollment. For a device-licensed, multi-tenant product (CLAUDE.md), decommissioning a device must release the license seat and destroy local keys/certs. Nothing in any doc addresses GDPR-style data deletion ('right to erasure') or crypto-shred of backups when a customer leaves.
- **Impact:** Without offboarding: license seats leak (customers pay for dead devices, or hit quota falsely), uninstalled machines leave plaintext-adjacent secrets and a still-valid agent cert on disk (stolen-laptop -> rogue agent), and the company cannot satisfy GDPR deletion requests from EU SMBs/hotels — a compliance and contractual liability. Decommission semantics shape what the Sprint 1 installer/uninstaller writes and wipes; bolting it on later means re-releasing the installer to every customer.
- **Recommendation:** Specify a decommission lifecycle now and implement the local half in Sprint 1's installer: uninstall must stop the service, securely delete keys/certs/config, and best-effort call a de-enrollment endpoint to release the license seat. Reserve a server-side 'revoke device' path (even if stubbed) so a stolen device's cert can be invalidated. Define crypto-shred as the GDPR deletion mechanism (destroy the wrapping key => data is unrecoverable) and document the retention/erasure policy. Ensure the installer writes secrets in a location/format the uninstaller can fully purge.

### GAP-5 — Heartbeat/poll protocol lacks versioning, transport security, retry/backoff, and clock-skew handling
`High severity · High likelihood · **S1**`

- **Risk:** PRD reqs 9-10 add a heartbeat sender and task-polling placeholder, and ARCHITECTURE lists Monitoring/Reporting on the SaaS. But there is no API contract or versioning scheme (no '/v1/' or capability negotiation), no defined transport auth (mTLS is deferred to Sprint 8 in ROADMAP, leaving Sprint 1 agents with unclear auth), no retry/backoff/jitter policy for when the SaaS is unreachable, and no time-sync/clock-skew tolerance — even though enrollment tokens, cert validity, signed-manifest freshness, and audit-log ordering all depend on agent time being trustworthy.
- **Impact:** No API versioning means the first protocol change breaks every deployed agent or forces a fleet-wide forced update (which itself depends on the still-unspecified update channel). No backoff/jitter means a brief SaaS outage triggers a thundering-herd of thousands of agents hammering the API on recovery, turning a blip into an outage. Clock skew on an agent makes 'single-use, time-limited' tokens and cert/expiry checks misbehave, and makes audit timestamps unreliable for forensics.
- **Recommendation:** Define a versioned API surface (URL or header version + capability flags) for heartbeat/poll from the first commit. Mandate TLS with server-cert validation in Sprint 1 and document the planned upgrade to mTLS so the cert store is forward-compatible. Specify exponential backoff with jitter and a max interval for all agent->SaaS calls. Require the agent to record both local and server-provided time, detect skew beyond a threshold, and log/heartbeat a clock-skew warning; use server time (or signed timestamps) for security-relevant decisions.

### GAP-6 — No RPO/RTO/SLA, retention, legal-hold, scheduling, or bandwidth-throttling requirements anywhere
`High severity · High likelihood · **S1**`

- **Risk:** This is a backup product whose buyers (SMBs, hotels, MSPs, multi-location companies) choose vendors on recoverability guarantees — yet no document states any RPO/RTO target, backup SLA, scheduling/policy model, retention/GRandfather-Father-Son scheme, legal-hold support, or bandwidth throttling. ARCHITECTURE mentions 'Backup Policies' as one SaaS bullet and ROADMAP puts 'Retention Policies' in Sprint 5, but there is no specification of the policy/schedule data model or any non-functional target. Bandwidth control (critical when backing up over a hotel's or branch office's thin uplink) is absent.
- **Impact:** Without RPO/RTO/SLA targets there is no objective definition of success and no basis for capacity planning, support commitments, or contracts; the product can ship and still 'fail' from the customer's perspective. Without bandwidth throttling, the first full backup saturates a branch office's WAN link and takes down the customer's business operations (the classic backup-vendor support escalation). Missing legal-hold/retention blocks MSP and regulated-SMB sales. The scheduling/policy data model influences the task-polling contract being stubbed in Sprint 1.
- **Recommendation:** Add a non-functional requirements section with explicit RPO/RTO and backup-success-SLA targets per tier, and capacity assumptions (devices/tenant, GB/device, concurrent uploads). Sketch the backup-policy/schedule data model now so the Sprint 1 task-polling placeholder returns a forward-compatible shape. Make bandwidth throttling (and a schedule window) first-class config fields in the Sprint 1 config.yaml schema even if enforcement lands in Sprint 3. Add retention and legal-hold to the policy model spec before Sprint 5.

### GAP-7 — Audit log is specified contradictorily and lacks integrity/tamper-evidence design
`High severity · Medium likelihood · **S1**`

- **Risk:** Audit logging is treated inconsistently: PRD security list and SECURITY.md require logging of enrollment/update/restore/integrity-fail/auth-fail events and PRD lists 'Audit log' and 'Tamper detection' as core security features; CLAUDE.md mandates structured logging in Sprint 1; but ROADMAP defers 'Audit Logging' to Sprint 8 and SECURITY.md lists 'Audit Trail' as a future feature. No spec exists for log schema, where audit events go (local file vs. SaaS), tamper-evidence (the docs demand 'tamper detection' but propose no hash-chaining/append-only/signing), retention, or PII redaction.
- **Impact:** A backup product is a prime ransomware target; attackers routinely wipe local logs to cover deletion of backups. A plain local log file with no tamper-evidence provides no forensic value precisely when it matters most. The Sprint-1-vs-Sprint-8 contradiction means the foundational structured-logging implementation built now will likely omit the fields, IDs, and append-only/chaining properties a real audit trail needs, forcing a rewrite. Unredacted logs of file paths/usernames create a GDPR exposure.
- **Recommendation:** Resolve the contradiction: design the audit-event schema (event type, actor, agent ID, tenant ID, monotonic sequence, server-anchored timestamp) as part of Sprint 1 structured logging, and emit security events to it from day one even if SaaS-side aggregation lands in Sprint 8. Make local audit entries tamper-evident (hash-chain or per-batch signature) to satisfy the docs' own 'tamper detection' requirement. Define log retention and a PII-redaction rule for paths/usernames. Mirror critical security events to the SaaS so a local wipe doesn't erase the trail.

### GAP-8 — No network reality support: proxy, IPv6, firewall/egress, or offline operation
`Medium severity · Medium likelihood · **S1**`

- **Risk:** SMB/hotel/branch-office networks routinely sit behind authenticating HTTP proxies, captive portals, strict egress firewalls, and increasingly IPv6-only or dual-stack networks. Nothing in any document addresses proxy configuration (PAC/auto-detect/manual creds), IPv6, the exact egress endpoints/ports the agent needs, TLS interception (corporate MITM proxies break cert pinning), or graceful offline/degraded operation when the SaaS is unreachable. The Sprint 1 agent makes outbound enrollment/heartbeat/poll calls into exactly these environments.
- **Impact:** Agents silently fail to enroll or heartbeat behind a corporate proxy — the single most common real-world deployment blocker for endpoint agents — generating heavy support load and stalled rollouts in the target SMB/hotel market. TLS-intercepting proxies will break naive cert validation/pinning (interacts with GAP-5). No documented egress endpoint list makes customer firewall whitelisting impossible. Proxy handling is an agent-core concern that belongs in the Sprint 1 config schema and HTTP client.
- **Recommendation:** Add proxy support (system proxy detection + explicit proxy host/port/credentials in config.yaml) to the Sprint 1 agent HTTP client, and decide the TLS-interception stance (allow a configured corporate CA, or document that intercepting proxies must whitelist the agent's endpoints). Verify IPv6/dual-stack works in the Go HTTP client. Publish the fixed list of egress FQDNs/ports for firewall whitelisting. Specify graceful-degraded behavior (queue heartbeats, backoff, continue local operation) when the SaaS is unreachable.

### GAP-9 — No observability, diagnostics bundle, or error-reporting strategy for a fleet of headless agents
`Medium severity · Medium likelihood · **S1**`

- **Risk:** The agent runs as a headless Windows Service across potentially thousands of devices, yet no document specifies metrics/telemetry, health/self-test endpoints, crash/error reporting back to the SaaS, or a support diagnostics-bundle mechanism (collect logs+config+versions for a ticket). CLAUDE.md asks only for 'structured logging'; ARCHITECTURE lists 'Monitoring/Reporting' as SaaS bullets with no agent-side counterpart. There is no defined way to know an agent is unhealthy beyond the bare heartbeat, and no way for support to pull diagnostics from a customer machine.
- **Impact:** Operating a backup fleet blind is untenable: silent backup failures (the worst kind for a backup product — customer discovers it only when restore fails) go undetected, support cannot diagnose field issues without a structured diagnostics bundle, and there's no fleet-wide health signal for the SaaS to alert on. Telemetry/diagnostics formats and the heartbeat health payload are agent-core and shape the Sprint 1 heartbeat schema.
- **Recommendation:** Extend the Sprint 1 heartbeat payload to carry health/status fields (agent version, last-error, service state, disk/connectivity status) so the SaaS has a real health signal, not just liveness. Add a local diagnostics-bundle command/flag (zip logs + redacted config + versions + system info) for support tickets now, while logging is being built. Decide an error/crash-reporting destination and PII-redaction policy. Reserve a metrics surface (even if just structured log counters initially) to feed future SaaS monitoring/alerting.

## 10. Architect-Added Risks (from the completeness review)

These risk classes were **not** in the original nine-dimension sweep but are equally Sprint-1-irreversible; surfaced by an adversarial completeness pass over the design.

### REV-1 — First-installed-binary supply-chain integrity is undefined (no SBOM, dep pinning, or vuln scan)
`High severity · Medium likelihood · **S1**`

- **Risk:** The whole risk set treats the *update*-signing key as the crown jewel but never asks how the *first* installed binary is trustworthy. ~10 third-party deps (zerolog, koanf, kardianos/service, zeebo/blake3, billgraziano/dpapi, cenkalti/backoff, oapi-codegen, google/uuid, lumberjack, jsonschema) ship inside a SYSTEM-level binary with no vetting policy. A compromised dependency or build runner owns the fleet exactly like a bad update; Authenticode alone does not address it.
- **Impact:** Supply-chain compromise of the agent at build time = SYSTEM-level RCE on every endpoint, indistinguishable from a malicious update, and not caught by the Ed25519 update-signature controls.
- **Recommendation:** Pin go.sum; set GOFLAGS=-mod=readonly; add govulncheck + gosec + gitleaks as REQUIRED CI gates; generate an SBOM (cyclonedx-gomod); vet and minimise the dependency set; build releases on a hardened, reproducible runner with provenance.

### REV-2 — Pinned-cert / embedded-key / agent-cert EXPIRY is an unrecoverable fleet-kill with no out-of-band path
`High severity · Medium likelihood · **S1**`

- **Risk:** SPKI pinning, a compiled-in Ed25519 key, and a finite-lifetime agent cert each have an expiry/rotation event. Nothing defines a recovery path that does not require re-distributing a new installer. A pinned cert that expires with no out-of-band re-pin = self-inflicted fleet outage (the classic pinning footgun).
- **Impact:** On pin/key/cert expiry with no rotation path, the entire fleet loses the control channel and/or update channel simultaneously and cannot self-recover — a mass outage requiring manual re-install on every device.
- **Recommendation:** Design rotation BEFORE first ship: pin/sign a SET (not one) with overlapping validity; deliver rotation material over the already-trusted channel ahead of expiry; document the installer-driven re-pin fallback; wire agent-cert renewal via the register endpoint (see Technical Design §0.2/§0.5).

### REV-3 — Agent concurrency / single-instance / bbolt single-writer races are unaddressed
`Medium severity · Medium likelihood · **S1**`

- **Risk:** Two agent processes (console + service), or the agent and a future updater, can race on the state store. bbolt is single-writer; the original design even placed updater FSM state inside the agent's bbolt store AND in a standalone JSON file — a concurrency hazard.
- **Impact:** State-store corruption, lost enrollment/credentials, or a hung service from lock contention; non-deterministic field failures that are hard to reproduce.
- **Recommendation:** Add a single-instance guard (named mutex / lock file) to the agent; make the agent the sole bbolt writer; give the updater its OWN state file (updater_state.json). Reconciled in Technical Design §0.1/§0.7.

### REV-4 — Installer/uninstaller is an unmodelled attack and data-loss surface
`Medium severity · Medium likelihood · **S1**`

- **Risk:** GAP-4 covers decommission but not: (a) the /TOKEN= one-shot file being readable in the window before first-run deletes it; (b) downgrade/reinstall-over-newer via the INSTALLER (anti-rollback is specified only for the updater path); (c) DLL/binary planting in Program Files if ACLs are not set atomically at create time.
- **Impact:** Local privilege escalation (binary planting), single-use-token theft, or a downgrade attack routed through the installer instead of the (protected) updater path.
- **Recommendation:** Set ACLs at folder-create time (not after); store the enrollment token SYSTEM-only and delete on consume; apply anti-rollback / version-floor checks on the installer path too, not just the updater; Authenticode-sign the installer.

### REV-5 — Data residency / tenant region is not a schema field, yet the market implies it
`Medium severity · Low likelihood · **S1**`

- **Risk:** 'Hotels / MSPs / multi-location companies' implies data-residency and tenant-region pinning, which must be a field in the enrollment/tenant record and ideally bound into the issued cert from day one. It is absent everywhere.
- **Impact:** Retrofitting a region/residency claim into already-issued certs and tenant records after a fleet exists is exactly the irreversible, flag-day-requiring change the rest of the review warns against; potential compliance (GDPR) exposure.
- **Recommendation:** Add a region/residency claim to the enrollment payload and tenant record now (even if a single default region in Sprint 1); reserve the cert claim so storage-region routing in Sprint 7 needs no re-enrollment.

### REV-6 — Contract-mock (Prism/Schemathesis) gives false confidence — it proves self-consistency, not Sprint-2 parity
`Medium severity · Medium likelihood`

- **Risk:** A mock generated from a self-authored OpenAPI spec validates only that the agent matches the spec — not that Sprint 2's FastAPI will. Single-use token enforcement, atomic consumption, and tenant authorization are BEHAVIORS a schema mock cannot represent; the spec itself can be wrong and every test still passes.
- **Impact:** Green Sprint-1 contract tests are mistaken for integration proof; semantic divergence surfaces only when the real backend lands in Sprint 2, after the agent contract is frozen and shipped.
- **Recommendation:** Treat openapi.yaml as a SHARED contract reviewed and co-owned by whoever builds Sprint 2; document the behaviors the mock cannot test; add server-side contract tests in Sprint 2; never report mock-green as 'integration verified'.
